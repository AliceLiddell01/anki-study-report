## 1. Краткий итог

C1.5R успешно завершён как полный исправляющий контур для ранее отклонённого пользовательского результата C1.5.

Последовательно закрыты:

1. восстановление корректного исходного состояния и статусов;
2. единая идентичность отображения карточки;
3. декларативный runtime компактного форматтера;
4. корректная семантика предпросмотра лицевой и обратной стороны;
5. независимые ограниченные источники кандидатов;
6. новая очередь карточек, требующих внимания;
7. пошаговый UX Inspection Profiles;
8. комплексная проверка всего контура и отдельная продуктовая приёмка владельцем.

Финальный кандидат прошёл профильную регрессионную проверку, канонический набор без Docker, сборку и проверку пакета и чистый полный real-Anki E2E с перезапуском Anki. Во время приёмки обнаружены и исправлены два рассогласования только в E2E-harness.

Production-поведение, публичные контракты payload и границы безопасности не ослаблялись.

PR #55 влит в `core`. В `master` ничего не переносилось. Release, deployment, публикация `.ankiaddon`, AnkiWeb и C1.6 не выполнялись.

## 2. Исходная ситуация

До возобновления завершения существовал один сохранённый кандидат:

- ветка: `c1-5r-7-integrated-acceptance-v2`;
- приостановленный кандидат: `dce5b3c4a4f16f213ca565e46f57ef4465415f20`;
- PR: #55;
- ранее зафиксированный `core`: `cc35dc626687948ba25c1e248faf39763fbfe523`;
- лучший сохранённый запуск GitHub Actions: `29828336280`.

В сохранённом запуске успешно прошли:

- каноническая проверка без Docker;
- package и provenance;
- round-trip артефакта;
- проверка гигиены кандидата;
- неизменяемый образ GHCR;
- целевой real-Anki E2E для Cards.

Полный real-Anki E2E завершился ошибкой в Inspection Profiles. Артефакты показали, что целевой scope `cards` успешно подготавливал профиль, а scope `full` оставлял его ненастроенным. Это указывало не на дефект продукта, а на рассогласование между подготовкой и использованием scope в E2E.

## 9. Выводы по процессу

Основная техническая часть успешно завершена, но путь оказался заметно длиннее необходимого из-за сочетания:

- старой временной инфраструктуры GitHub Actions;
- неполного tooling WSL;
- различий PATH Windows и Linux;
- неподдерживаемой версии Python;
- сети Docker bridge;
- нескольких защитных ошибок в scripts завершения.

Положительный результат состоит в том, что защитные проверки почти всегда останавливались до опасных действий. Ни одна промежуточная ошибка не привела к:

- force-push;
- потере несвязанной истории;
- изменению `master`;
- ослаблению production-контрактов;
- коммиту сгенерированных артефактов;
- ложному объявлению ошибки harness дефектом продукта.

Для следующих этапов следует применять закреплённые правила работы с репозиторием:

1. один этап — одна рабочая ветка и один основной PR;
2. сначала проверка окружения;
3. Python строго по `.python-version`;
4. нативные Linux-инструменты в WSL без затенения командами Windows;
5. Docker Compose и Buildx проверяются до E2E;
6. полный real-Anki E2E используется как интеграционный gate, а не debugger;
7. временные controller- и trigger-workflows не создаются для обычных операций Git;
8. после одного подтверждённого PASS gate того же SHA не повторяется без новой причины.
