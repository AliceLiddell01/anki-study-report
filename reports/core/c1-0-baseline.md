## Политика долгоживущей ветки и поставки

`core` — независимая долгоживущая ветка для последовательной работы над C1 и C2.

- Не создавать pull request в `master`, не выполнять merge, tag, публикацию GitHub Release, `.ankiaddon`, deployment или публикацию на AnkiWeb, пока владелец отдельно не одобрит стабильную сборку Core.
- Не закрывать весь трек Core после одной части C1.
- Синхронизироваться с `master` только по явно названной причине и проверять каждое перенесённое изменение.
- Не выполнять автоматически merge или rebase несвязанной работы.
- Не использовать force-push без отдельного одобрения владельца.
- Сообщения коммитов должны описывать фактическое изменение.
- До явного изменения политики поставки изменения этого трека отправляются только в `core`.

## Инвентаризация существующих зависимостей C1

### Cards

- Канонические ключи payload: `attentionCards`, `attentionCardsStatus` и `noteTypeCatalog`.
- Актуальный collector работает только на чтение, возвращает не более 100 карточек и публикует диагностику периода, фильтрации колод, профилей заметок и количества проблем.
- Текущие семейства проблем включают проблемы обучения (`leech`, `repeated_again`, `slow_answer`, `low_pass_rate`) и эвристические проблемы содержимого (`missing_audio`, `missing_example`, `missing_image`, `missing_meaning`, `missing_part_of_speech`).
- Текущая сортировка в основном опирается на непрозрачный числовой `riskScore`.
- Cards v1 отвечает за фильтры, передачу в Browser и три режима отображения, но не содержит полноценного editor.

### Search

- Существующие `POST /api/search/query` и `/api/search/inspect` защищены токеном, ограничены, строго валидируются и работают только на чтение.
- Нативная грамматика поиска Anki и структурированные фильтры используются без произвольного SQL или второго Search API.
- `useSearchWorkspace` уже отвечает за отмену по latest-wins, pagination, явный выбор до 200 элементов, состояние Inspector и обновление с reconciliation после действий.
- C1 должен переиспользовать это рабочее пространство и его путь inspect, а не дублировать состояние запросов.

### Safe Actions

- Действия с карточками ограничены suspend/unsuspend, set/clear flag, bury/unbury и перемещением в обычную колоду.
- Действия с заметками ограничены добавлением и удалением tags.
- Запросы используют точные поля, положительные десятичные строковые ID, ограничение пакета 200 элементов и полную предварительную проверку до mutation.
- Устаревшая сущность отклоняет весь пакет. Результаты no-op имеют явный тип и не создают undo; mutations используют существующий официальный bridge операций Anki и один шаг undo.
- До отдельного пересмотра контракта C1 может использовать только эти существующие действия.

### Signals

- Signals локальны, ограничены и не изменяют collection.
- Текущий registry детекторов содержит давление нагрузки, недавнее падение retention, ухудшение состояния колоды и repeated Again.
- `card.repeated_again` основан на одном сгруппированном запросе revlog за семь дней, не возвращает содержимое карточки и ограничен 50 карточками.
- Устранение определяется детектором и требует двух успешных оценок без кандидата; состояние прочтения не зависит от устранения сигнала.
- C1 должен переиспользовать жизненный цикл сигналов, а не добавлять ручное скрытое или resolved-состояние.

### Передача из Notification

- Notification Center является локальным для профиля и не является inbox аккаунта.
- Контекст карточки или колоды передаётся через строгий механизм только для сессии с точными полями, десятичными ID сущностей, максимальным возрастом 10 минут и однократным потреблением.
- ID сущностей не помещаются в hash, постоянное локальное хранилище или удалённую телеметрию.
- C1 может потреблять эту передачу, но не должен создавать второй контракт навигации уведомлений.

### Изоляция предпросмотра

- HTML, CSS и media предпросмотра санитизируются на backend.
- Режимы `table` и `tiles` используют предпросмотр только лицевой стороны в Shadow DOM.
- `ankiPreview` использует один host Shadow DOM только для ответа из санитизированного `renderedPreview.backHtml`.
- Iframe и выполнение JavaScript шаблона запрещены.
- Media остаются защищёнными токеном и проходят проверку имени файла.

## Известные продуктовые выводы по Cards v1

Это принятые выводы исходного состояния, а не реализация C1.1.

1. `risk` и `check` возвращают все текущие отфильтрованные строки; `check` лишь дополнительно требует выбрать колоду.
2. `gaps` фильтрует проблемы отсутствующего содержимого.
3. `patterns` фильтрует `repeated_again`, `slow_answer` и `low_pass_rate`.
4. Подписи Risk, Problems/Patterns и Check не образуют отдельные пользовательские workflows и частично перекрываются.
5. `table`, `tiles` и `ankiPreview` показывают в основном одну очередь с разной плотностью и могут создавать очень длинные страницы.
6. Предварительное направление C1: одна очередь Requires attention, фильтры вместо похожих на workflows вкладок, компактная очередь с Inspector и явные причины и подтверждения вместо непрозрачной оценки.
7. Полное редактирование остаётся в Anki Browser.
8. Проверки качества содержимого требуют подтверждённый профиль каждого типа заметки. Текущая реализация эвристически определяет роли и виды; до признания отсутствующего содержимого авторитетной проблемой C1.1 должен определить явный контракт подтверждения и fail-closed-поведение.
9. Универсальные проблемы обучения не зависят от Inspection Profile.

## Границы безопасности

- Frontend никогда не обращается к collection Anki напрямую.
- Server остаётся привязанным к `127.0.0.1`, а все чувствительные вызовы API требуют токен dashboard.
- Проверку токена, редактирование чувствительных данных и ограничения артефактов с токеном нельзя ослаблять.
- Не создаются дублирующий Search API, второй стек действий или второй жизненный цикл сигналов.
- Запрещены произвольные SQL, RPC, JavaScript, Python и iframe.
- Sanitizer, проверка имён media-файлов и изоляция предпросмотра Shadow DOM обязательны.
- Runtime-данные, логи, скриншоты, caches, данные профиля, токены, сгенерированные assets, `.ankiaddon` и результаты E2E остаются вне Git.
- Подтверждения Signals и ID сущностей остаются локальными и не входят в taxonomy удалённой телеметрии.

## Пути проверки

### Документальный путь C1.0

Применимый минимум: проверка metadata репозитория, ветки и head, изучение изменённых файлов и проверка ссылок и путей Markdown. Fast CI и Docker E2E не требуются, поскольку C1.0 меняет только документацию.

### Fast CI

Workflow: **Fast CI** (`.github/workflows/ci-fast.yml`).

- `workflow_dispatch` включён и не имеет пользовательских inputs.
- Автоматические triggers push и PR нацелены на `master`, поэтому push в `core` не запускает его автоматически.
- В UI GitHub Actions необходимо выбрать ветку `core` в **Run workflow**.
- Ручной запуск для ветки не по умолчанию сравнивается с `origin/master` через planner проверки.

### Real-Anki Docker E2E

Workflow: **Full Docker / Anki E2E** (`.github/workflows/ci-e2e.yml`).

Ручные inputs:

- `mode`: `standard`, `strict-apkg`, `perf100`;
- `scope`: `full`, `global`, `stats`, `decks`, `activity`, `cards`, `settings`, `notifications`;
- `screenshot_workers`: `auto`, `1`, `2`, `3`, `4`;
- `resource_telemetry`: boolean;
- `verify_restart`: `auto`, `true`, `false`;
- `fast_ci_run_id`: ID точного успешного запуска Fast CI.

Workflow можно запустить с `core` в качестве ref. Cloud E2E отклоняет fallback со сборкой из исходников, поэтому обычный запуск ветки Core должен использовать точный успешный пакет Fast CI через `fast_ci_run_id`. C1.0 не требует этого запуска. Будущие изменения рендера или media Cards требуют целевого browser-scope Cards и, при необходимости по матрице проверок, финального расширения проверки.

## Фактически выполненные проверки C1.0

- проверены идентичность репозитория и разрешения;
- подтверждено, что ветка по умолчанию — `master`;
- проверены SHA и сообщение актуального исходного коммита;
- до создания подтверждено отсутствие `core` и других очевидных веток C1, Cards или triage;
- подтверждено отсутствие открытых pull requests;
- `core` создана непосредственно из проверенного исходного SHA;
- изучены обязательные актуальные документы, контракты, workflows и репрезентативные production-пути и тесты;
- scope изменений ограничен файлами документации и отчёта;
- после коммита ветка и изменённые файлы прочитаны повторно;
- сравнение с `master` подтвердило расхождение только в документации.

## Невыполненные проверки

- локальный `git status --short --branch`;
- локальный `git rev-parse HEAD`;
- локальный `git branch --show-current`;
- локальный `git log -1 --oneline`;
- локальный `git diff --check`;
- scripts проекта, Python-тесты, frontend-тесты, typecheck или проверка пакета;
- Fast CI;
- Docker и real-Anki E2E;
- проверка скриншотов Cards из `cards.zip`.

Ни одна невыполненная команда не заявляется как прошедшая.

## Ручная проверка как запасной путь

Для принятия этого документального исходного состояния C1.0 ручная проверка не требуется.

Для будущего изменения runtime или UI в C1.1:

1. Открыть GitHub **Actions → Fast CI → Run workflow**.
2. Выбрать ветку `core` и запустить workflow.
3. Зафиксировать ID успешного запуска для точного коммита Core.
4. Если матрица проверок требует real Anki, открыть **Actions → Full Docker / Anki E2E → Run workflow**.
5. Выбрать ветку `core`, подходящие риску mode и scope и передать успешный `fast_ci_run_id`.
6. До утверждения PASS изучить сводку запуска и отредактированные артефакты.

## Риски долгоживущей ветки Core

| Риск | Обязательный контроль |
| --- | --- |
| Расхождение с `master` | периодически проверять коммиты; переносить только явно названные исправления с отдельной проверкой |
| Исправления попадают только в `master` | оценивать применимость каждого исправления к Core; выполнять cherry-pick или merge осознанно, а не автоматически |
| Исправления Core нужны в `master` | не вливать всю ветку; извлекать только отдельно одобренные и проверенные изменения |
| Устаревание контрактов во время длинного цикла C1/C2 | обновлять исходное состояние и передачу контекста при изменении публичного контракта или политики ветки |
| Эвристические проверки содержимого | до авторитетного Triage требовать подтверждённые профили типов заметок и fail-closed-поведение |
| Отсутствие подтверждений скриншотами | изучить светлые и тёмные APKG и синтетические фикстуры после появления `cards.zip` или эквивалентного подтверждения |
| В этой сессии нет проверки локального рабочего дерева | опираться только на объекты GitHub и compare; при наличии checkout выполнить локальные проверки |
| Чрезмерное использование тяжёлых проверок | следовать матрице тестирования и запускать Docker только для риска, который нельзя покрыть unit-тестами или metadata |

## Точные предварительные условия C1.1

1. Продолжать работу в `core`; не создавать PR, не выполнять merge или выпуск.
2. Считать этот отчёт и актуальные production-контракты исходным состоянием.
3. Определить продуктовый контракт одной канонической очереди Triage: идентичность элемента, источник, причина, серьёзность, подтверждение, статус и детерминированный порядок.
4. Определить, как `attentionCards`, активные Signals карточек и явная передача из Search и Notification входят в эту очередь без дублирующего хранилища или API.
5. Определить жизненный цикл подтверждения Inspection Profile и fail-closed-поведение проверок качества содержимого.
6. Сохранить независимость универсальных проблем обучения от профилей типов заметок.
7. Переиспользовать Search inspect, Safe Actions, передачу Notification и изолированный предпросмотр.
8. До реализации совместно описать изменения backend, frontend, types, tests и docs.
9. Выбрать профильные, Fast CI и real-Anki проверки по фактическому будущему diff.
