# Anki Review Event Taxonomy

Статус: **DRAFT v0.1**  
Дата: **2026-07-16**  
Область: **первый этап проектирования Review XP; продуктовый концепт, без реализации в production-коде**

## 1. Назначение документа

Этот документ определяет, какие события повторения в Anki считаются самостоятельной полезной учебной работой и могут участвовать в расчёте `Review XP`.

Он отвечает на вопрос:

> Что именно является награждаемой единицей повторения до применения формул FSRS, временных коэффициентов, объёмных бонусов и дневных ограничений?

Документ является детализацией раздела `Review XP` из [anki-xp-foundation.md](anki-xp-foundation.md). Общая прогрессия, уровни, стрик и `Momentum` описаны в [progression-foundation.md](progression-foundation.md).

На этом этапе не назначаются окончательные числовые значения XP. Сначала необходимо отделить реальную проверку памяти от вспомогательных, административных, повторных и потенциально эксплуатируемых событий.

## 2. Основной принцип

Единицей награждения является не строка `revlog`, не показ карточки и не одно нажатие кнопки.

Основная единица:

```text
Review Episode
=
первичная попытка воспроизвести уже изученную карточку
+
связанные с ней шаги восстановления после ошибки
```

Примеры:

```text
Due-карточка → Good
= один успешный Review Episode

Due-карточка → Again → Good через 10 минут
= один неуспешный первичный Review Episode
  + один вспомогательный relearning event

Due-карточка → Again → Again → Good
= всё ещё один Review Episode,
  а не три независимых источника полного XP
```

Причины использовать `Review Episode`:

- одна карточка не должна приносить полный XP несколько раз за день;
- число строк `revlog` не должно автоматически означать объём независимой работы памяти;
- `Again` и последующее relearning должны оставаться связанными;
- внутридневные шаги восстановления должны учитываться, но не превращаться в farm;
- расчёт XP должен быть объясним пользователю как единый эпизод;
- отмена исходного ответа должна отменять и связанную награду.

## 3. Базовый инвариант дня

```text
Не более одного основного Review Episode
на одну card ID за один учебный день Anki.
```

Учебный день определяется границей дня Anki, а не обязательной календарной полуночью операционной системы.

Следствия:

- первая подходящая попытка карточки может получить класс `core`;
- relearning-показы остаются внутри того же эпизода;
- дополнительные ранние или ручные показы не создают новый `core`-эпизод;
- переход через границу дня требует отдельной связи с родительским эпизодом;
- удаление и повторное создание карточки не должно автоматически обходить lifetime-защиту.

Точная техническая реконструкция учебного дня относится к будущей реализации.

## 4. Граница между Review XP и Learn XP

Для игровой экономики нельзя считать все строки истории единым видом review.

Предлагаемое разделение:

| Состояние карточки до ответа | Канал XP |
|---|---|
| Новая карточка | `Learn XP` |
| Карточка на initial learning steps | `Learn XP` |
| Interday learning до выпуска в review | `Learn XP` |
| Карточка в нормальном review-состоянии | `Review XP` |
| Первый `Again` на review-карточке | `Review XP`, неуспешный исход |
| Relearning после lapse | Вспомогательная часть `Review XP` |
| Ранний показ review-карточки | Supplemental Review |
| Preview без изменения расписания | Practice-only или `none` |
| Создание и редактирование заметки | `Create XP` |

### 4.1. Принятое решение

`Relearning` не относится к изучению нового материала.

Это восстановление ранее изученной карточки, поэтому оно остаётся частью Review-домена, но не создаёт новое полноценное событие `core`.

## 5. Многомерная классификация

Одно событие не следует кодировать одним перегруженным enum. Оно классифицируется по независимым осям.

### 5.1. Состояние карточки

```text
new
learning
review
relearning
unknown
```

### 5.2. Отношение к естественному сроку

```text
on_time
  Карточка была естественно доступна для планового повторения.

overdue
  Естественный срок уже прошёл.

early
  Естественный срок ещё не наступил.

forced_due
  Карточка была искусственно сделана due ручной или массовой операцией.

unknown
  Исходное происхождение срока нельзя надёжно определить.
```

### 5.3. Источник предъявления

```text
normal_queue
filtered_rescheduling
filtered_preview
review_ahead
manual_operation
imported_history
custom_scheduler
unknown
```

### 5.4. Роль внутри Review Episode

```text
primary
  Исходная попытка воспроизведения.

recovery
  Relearning после первичного Again.

supplemental
  Дополнительная практика, не равная плановой проверке памяти.

duplicate
  Повторное предъявление без самостоятельной награждаемой ценности.

administrative
  Служебная операция, а не учебная попытка.
```

### 5.5. Исход ответа

```text
fail
pass_hard
pass_good
pass_easy
none
unknown
```

Для классификации:

- `Again` означает неуспешное воспроизведение;
- `Hard`, `Good` и `Easy` являются успешными исходами;
- разница между успешными кнопками не определяет автоматически размер игрового XP;
- кнопка остаётся пользовательской оценкой памяти для scheduler, а не выбором игровой награды.

### 5.6. Класс награждения

```text
core
  Основное событие, к которому применяется Review Reward Model.

support
  Вспомогательная работа внутри исходного эпизода с отдельным cap.

supplemental
  Дополнительная практика с жёстко ограниченной отдачей.

none
  Постоянный XP не начисляется.

route_to_learn
  Событие передаётся в Learn XP.

deferred
  Класс нельзя назначить без дополнительной информации о происхождении.
```

## 6. Сводная матрица

| Код | Событие | Класс | Основной смысл |
|---|---|---|---|
| `R1` | Первое естественное due-review | `core` | Эталонный Review Episode |
| `R2` | Первое overdue-review | `core` | Реальная работа без неограниченного бонуса за просрочку |
| `R3` | Due-review в filtered deck с rescheduling | `core` | Эквивалент обычного due-review |
| `R4` | Первичный `Again` | `core` / `fail` | Честная неуспешная попытка |
| `R5` | Intraday relearning | `support` | Восстановление после lapse |
| `R6` | Interday relearning | `support` | Продолжение предыдущего lapse-эпизода |
| `R7` | Early review | `supplemental` | Дополнительная практика до срока |
| `R8` | Filtered preview без rescheduling | `none` для постоянного XP | Практика без изменения учебного состояния |
| `R9` | Повторный показ той же карточки | `supplemental` или `none` | Защита от повторного farm |
| `R10` | Manual/reset/reschedule record | `none` | Административное событие |
| `R11` | Принудительно сделанная due-карточка | `supplemental` | Реальный ответ, но искусственно выбранный срок |
| `R12` | Массовое FSRS-rescheduling | `none` для служебной записи | Реальный следующий ответ классифицируется отдельно |
| `R13` | Отменённый ответ | `none` | Событие и награда отозваны |
| `R14` | New/learning event | `route_to_learn` | Относится к первоначальному освоению |
| `R15` | История удалённой карточки | сохраняет прошлый XP | Удаление не переписывает прошлую работу |
| `R16` | Sibling review | обычно `core` + метка | Отдельный prompt, возможная подсказка от sibling |
| `R17` | Review без надёжных FSRS-данных | `core` с нейтральной оценкой | Право на XP не зависит от зрелости модели |
| `R18` | Импортированная или старая история | `deferred` | Требуется отдельная политика provenance |

## 7. R1 — первое естественное due-review

### 7.1. Условия

Событие относится к `R1`, если:

- карточка находится в review-состоянии;
- естественный срок наступил;
- это первая основная попытка этой card ID за Anki-день;
- событие возникло из обычной очереди или эквивалентного режима;
- оно не было отменено;
- due-состояние не было искусственно создано непосредственно перед показом;
- происхождение события достаточно надёжно для классификации.

### 7.2. Классификация

```text
eligibility_class: core
episode_role: primary
due_relation: on_time
```

### 7.3. Исходы

```text
Again → core event с outcome=fail
Hard  → core event с outcome=pass_hard
Good  → core event с outcome=pass_good
Easy  → core event с outcome=pass_easy
```

Все четыре ответа относятся к одному типу первичного эпизода. Разница в будущем учитывается в `OutcomeFactor` и других сигналах, а не через право или отсутствие права на событие.

## 8. R2 — просроченное повторение

### 8.1. Условия

Карточка была естественно due ранее, но повторяется после срока.

```text
eligibility_class: core
episode_role: primary
due_relation: overdue
```

### 8.2. Принцип

Просрочка не делает реальное воспроизведение фиктивным. Пользователь выполняет настоящую работу памяти, поэтому базовое право на Review XP сохраняется.

Однако событие должно хранить контекст:

```text
scheduled_due_before
days_overdue
relative_overdue
retrievability_before
schedule_origin
```

### 8.3. Ограничение

Низкая retrievability или большая просрочка не гарантирует неограниченный бонус.

Иначе оптимальной игровой стратегией станет намеренное накопление backlog.

Следовательно:

- базовая награда сохраняется;
- challenge-бонус рассчитывается отдельно;
- экстремальная просрочка может получить cap или decay;
- административно созданная просрочка не равна естественной.

## 9. R3 — due-review в filtered deck с rescheduling

Если карточка уже была естественно due, а filtered deck сохраняет изменение расписания по ответам, такое событие считается эквивалентом обычного планового review.

```text
eligibility_class: core
source: filtered_rescheduling
due_relation: on_time | overdue
```

### 9.1. Принятое решение

```text
Естественно due в обычной очереди
=
Естественно due в filtered deck с rescheduling
```

Само расположение карточки в filtered deck не уменьшает награду.

## 10. R4 — первичный Again

### 10.1. Классификация

Если естественная review-карточка была предъявлена в подходящем `core`-контексте и пользователь выбрал `Again`:

```text
eligibility_class: core
episode_role: primary
outcome: fail
```

### 10.2. Почему XP не равен нулю

Пользователь:

- действительно проверил память;
- обнаружил забывание;
- потратил содержательное усилие;
- инициировал восстановление материала.

### 10.3. Ограничение

Полная награда предназначена для успешного воспроизведения. Поэтому `Again` получает пониженный `OutcomeFactor`, но не удаляется из экономики.

## 11. R5 — intraday relearning

Все показы review-карточки, возникшие после первичного `Again` в пределах того же учебного дня, относятся к родительскому эпизоду.

```text
eligibility_class: support
episode_role: recovery
parent_episode_id: required
```

Ограничения:

- не применяется полная формула Review XP;
- не выдаётся новый полный challenge-бонус;
- событие не увеличивает число уникальных Review Episodes;
- все recovery-события имеют общий cap на lapse-эпизод;
- повторные `Again` внутри relearning не создают бесконечный reward loop;
- возможен небольшой completion credit за успешное завершение relearning.

Точные величины относятся к Review Reward Model.

## 12. R6 — interday relearning

Если recovery step переносится через границу Anki-дня:

```text
eligibility_class: support
episode_role: recovery
parent_episode_id: previous lapse episode
```

Это реальная работа нового дня, но не новый полноценный `core`-эпизод той же проверки памяти.

Предварительное правило для будущего дневного агрегирования:

> Interday relearning может участвовать в полезном объёме дня, но один такой шаг не должен автоматически закрывать минимальный порог стрика.

## 13. R7 — раннее повторение

### 13.1. Определение

Review-карточка была показана до наступления естественного срока через filtered deck, Review Ahead или аналогичный механизм.

```text
eligibility_class: supplemental
episode_role: supplemental
due_relation: early
```

### 13.2. Почему не core

Пользователь сам выбирает момент повторения и может многократно возвращать лёгкие карточки раньше оптимального срока.

### 13.3. Почему не обязательно none

Ранняя практика может быть содержательной:

- перед экзаменом;
- перед поездкой;
- перед использованием материала;
- при целенаправленном повторении темы.

### 13.4. Ограничения

- учитывается не более одного раннего события card ID за день;
- нет полного challenge-бонуса;
- memory gain не учитывается в полном объёме;
- действует небольшой дневной cap;
- early review самостоятельно не закрывает стрик;
- повторные ранние показы получают `none`;
- естественный будущий due-review сохраняет право стать `core`, если наступит в другой день.

## 14. R8 — filtered preview без rescheduling

Если показ не изменяет учебное состояние и карточка возвращается к прежнему расписанию:

```text
eligibility_class: none для постоянного XP
source: filtered_preview
episode_role: supplemental
```

### 14.1. Рекомендация

Система может отдельно отображать практическую активность:

```text
Дополнительная практика: 42 карточки
```

Но такая активность не обязана конвертироваться в общий постоянный XP.

Причина: preview можно повторять неограниченно без естественного изменения состояния обучения.

## 15. R9 — повторный показ той же карточки

Если повторный показ не является обязательным recovery step:

```text
первый подходящий основной эпизод: core
первый осмысленный дополнительный показ: supplemental или none
последующие показы: none
```

Защищаемые сценарии:

- повторная пересборка filtered deck;
- многократный Review Ahead;
- ручное изменение срока;
- повторение одной лёгкой карточки ради XP;
- несколько одинаковых действий в пределах короткого времени.

## 16. R10 — ручные и административные операции

К категории относятся:

- `Reset`;
- `Set Due Date`;
- ручное reschedule;
- массовая смена сроков;
- служебная запись после изменения scheduler-параметров;
- изменение счётчиков без учебного ответа;
- другие операции, не содержащие попытку воспроизведения.

```text
eligibility_class: none
episode_role: administrative
```

### 16.1. Безусловный инвариант

> Административная запись никогда не приносит XP.

Она может менять будущую классификацию следующего реального ответа, но не является работой памяти сама по себе.

## 17. R11 — принудительно сделанная due-карточка

### 17.1. Сценарий

Будущей карточке вручную назначили текущий срок, после чего пользователь действительно попытался ответить.

Это реальная попытка, но момент её появления выбран административно.

```text
eligibility_class: supplemental
due_relation: forced_due
```

### 17.2. Правило

```text
Карточка уже была естественно due до операции
→ сохраняет core eligibility.

Карточка была будущей и стала due только после операции
→ supplemental до естественного срока.
```

### 17.3. Требуемый контекст

```text
natural_due_before_operation
manual_operation_timestamp
forced_due_value
was_naturally_due
```

Это предотвращает farm через массовое `Set Due Date: today`.

## 18. R12 — массовое FSRS-rescheduling

Нужно разделять:

```text
служебную запись reschedule
→ none

последующий реальный ответ
→ классифицируется как отдельное учебное событие
```

Если после изменения параметров карточка стала due и пользователь реально попытался её вспомнить, базовая работа памяти сохраняется.

Но событие получает метку:

```text
schedule_origin: bulk_fsrs_reschedule
```

Она необходима, чтобы позднее:

- не выдавать ложный bonus за административно созданную просрочку;
- не считать такую нагрузку естественным backlog;
- сохранить базовую награду за реальный ответ;
- анализировать влияние массового reschedule отдельно.

## 19. R13 — отменённый ответ

Если пользователь выполнил Undo учебного ответа:

```text
eligibility_class: none
undone: true
```

Следствия:

- provisional reward отзывается;
- recovery-события, порождённые отменённым ответом, не должны сохранять самостоятельный XP;
- агрегаты дня должны быть пересчитаны;
- отменённое событие может оставаться в техническом аудите, но не в reward ledger как окончательная награда.

Архитектурный вывод:

> Начисление может быть provisional до подтверждения, что исходное событие сохранилось.

## 20. R14 — new и learning events

```text
eligibility_class: route_to_learn
```

К ним относятся:

- первый показ новой карточки;
- initial learning steps;
- interday learning до выпуска карточки в review;
- повторные шаги первоначального освоения.

Они не участвуют в Review XP, даже если техническая история использует общий механизм записей ответов.

## 21. R15 — удалённая карточка

Если карточка была удалена после ранее подтверждённых review-событий:

```text
уже заработанный XP сохраняется;
удаление не переписывает прошлую работу;
история может потерять текущую deck attribution;
повторное создание похожего материала не получает старый XP автоматически.
```

Это следует общему инварианту постоянного прогресса: позднее изменение коллекции не конфискует подтверждённые усилия.

Для защиты от повторного создания потребуется отдельная lifetime/provenance-модель.

## 22. R16 — sibling-карточки

Разные карточки одной заметки могут проверять разные направления или факты, поэтому каждая card ID потенциально создаёт отдельный Review Episode.

```text
eligibility_class: обычно core
context_tag: sibling_exposed | sibling_not_exposed | unknown
```

### 22.1. Почему не обнулять автоматически

- front→back и back→front требуют разных retrieval acts;
- разные cloze могут проверять разные элементы;
- sibling не всегда раскрывает ответ другой карточки;
- модель заметки может создавать действительно независимые карточки.

### 22.2. Почему сохранять контекст

Ранее показанный sibling способен подсказать ответ. Поэтому Review Reward Model может ограничить только часть challenge-бонуса второго sibling, не отменяя базовую работу.

## 23. R17 — review без надёжных FSRS-данных

Право на базовый Review XP не зависит от наличия зрелой персонализированной модели FSRS.

```text
FSRS state доступно и надёжно
→ полная контекстная формула.

FSRS state доступно, но confidence низкий
→ ограниченные FSRS-бонусы.

FSRS выключен или state недоступно
→ нейтральный базовый Review XP без FSRS challenge bonus.
```

Требуемые поля:

```text
memory_model: fsrs | legacy | unavailable
memory_state_confidence: high | medium | low | unknown
```

Это предотвращает две крайности:

- новичок или пользователь legacy scheduler не лишается XP;
- неточная модель не выдаётся за абсолютно достоверную оценку сложности.

## 24. R18 — импортированная и старая история

Историческое событие получает provenance:

```text
live_verified
historical_native
historical_imported
historical_unknown
```

Для старой истории могут быть неизвестны:

- исходное содержимое карточки;
- состояние scheduler до ответа;
- параметры FSRS на тот момент;
- естественное происхождение due;
- наличие ручного reschedule;
- точная связь с устройством и синхронизацией;
- факт импорта или восстановления коллекции.

Поэтому:

```text
eligibility_class: deferred
```

Ретроспективное начисление проектируется отдельно и не должно автоматически использовать полную live-формулу.

## 27. Дедупликация

Минимальные правила:

1. Одна исходная `revlog`-запись не обрабатывается повторно.
2. Одна card ID получает не более одного `core` Review Episode за Anki-день.
3. Recovery events привязываются к родительскому lapse-эпизоду.
4. Повторная синхронизация не создаёт новый reward event.
5. Undo отзывает provisional reward.
6. Импортированная копия истории не должна удваивать уже учтённые события.
7. Reset карточки не сбрасывает lifetime reward state.
8. Удаление и повторное создание не гарантирует новую ретроспективную награду.

Точный `dedupe_key` проектируется после изучения фактических идентификаторов и поведения синхронизации Anki.

## 28. События, которые не могут самостоятельно закрыть стрик

До проектирования дневного агрегирования предварительно запрещено считать полноценным днём только:

- один preview без rescheduling;
- один early review;
- один повторный показ той же карточки;
- одну ручную операцию;
- одну служебную reschedule-запись;
- один interday recovery step;
- действие, классифицированное как `none`;
- событие с неопределённым или недостоверным происхождением без дополнительной проверки.

Это не означает, что вся такая работа всегда имеет нулевую аналитическую ценность. Она лишь не является самостоятельным доказательством полноценного учебного дня.

## 29. Explainability

Для каждого эпизода система в будущем должна уметь объяснить классификацию.

Примеры:

```text
Core review
Причина: первое естественное due-review карточки сегодня.

Supplemental review
Причина: карточка повторена раньше естественного срока.

Support credit
Причина: шаг relearning после исходного Again.

No persistent XP
Причина: preview не изменил состояние обучения.

No XP
Причина: запись создана ручным reschedule, а не ответом.
```

Пользователь не обязан видеть все технические поля, но итог не должен выглядеть случайным.

## 30. Предварительно принятые решения

1. Награждаемая единица — `Review Episode`, а не строка `revlog`.
2. На card ID допускается не более одного `core`-эпизода за Anki-день.
3. Естественные due и overdue reviews относятся к `core`.
4. Первичный `Again` остаётся `core`-событием с outcome `fail`.
5. Relearning является вспомогательной частью исходного lapse-эпизода.
6. New и initial learning полностью направляются в Learn XP.
7. Естественно due-review в filtered deck с rescheduling равно обычному review.
8. Early review не получает полную награду.
9. Preview без rescheduling не даёт постоянный Review XP.
10. Manual/reset/reschedule records никогда не дают XP.
11. Принудительно сделанная due-карточка не считается естественным due-review.
12. Undo отзывает связанную награду.
13. Удаление карточки не отнимает подтверждённый XP.
14. Sibling остаётся отдельной карточкой, но получает contextual tag.
15. FSRS влияет на качество расчёта, но не на базовое право получить XP.
16. Исторические события классифицируются по provenance отдельно.
17. События `support` и `supplemental` имеют собственные caps и не используют полную core-формулу.
18. Классификация должна быть объяснимой и аудируемой.

## 31. Неопределённые вопросы

Следующие вопросы намеренно остаются открытыми:

- точный размер `OutcomeFactor` для `Again`;
- размер support credit за relearning;
- допустимая награда за early review;
- нужен ли отдельный непостоянный Practice Credit;
- как именно sibling exposure изменяет reward;
- какая давность ручной операции перестаёт влиять на due provenance;
- как классифицировать custom scheduler add-ons;
- как реконструировать естественный due после массового reschedule;
- как связывать interday recovery с родительским episode;
- как учитывать события, созданные на нескольких устройствах до sync;
- как определять и обрабатывать повторно импортированную историю;
- какие поля доступны live и какие требуют снимка state before answer.

## 32. Следующий этап

Следующий документ:

```text
anki-review-reward-model.md
```

Он должен определить относительную ценность одного `core` Review Episode:

```text
BaseReviewReward
× OutcomeFactor
× RetrievalChallengeFactor
× MemoryGainFactor
× DifficultyFactor
× ResponseValidityFactor
× ContextFactor
```

Первой подзадачей будет проектирование функции:

```text
Retrievability before answer
→ Retrieval Challenge reward
```

Необходимо сравнить несколько форм кривой и проверить, чтобы она:

- повышала награду за трудное успешное воспроизведение;
- не делала намеренную просрочку выгодной;
- не наказывала возвращение после backlog;
- не создавала резкого скачка на границах диапазонов;
- корректно работала при низкой уверенности FSRS;
- оставалась объяснимой пользователю.
