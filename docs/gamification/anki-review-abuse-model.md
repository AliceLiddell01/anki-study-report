# Anki Review XP Abuse Model

Статус: **DRAFT v0.1**  
Дата: **2026-07-16**  
Область: **третий этап проектирования Review XP; продуктовая abuse-модель без production-реализации**

## 1. Назначение документа

Этот документ определяет, как Review XP должен противостоять повторному начислению, простому фарму, манипуляциям scheduler и другим стратегиям, при которых игровой прогресс растёт быстрее реальной учебной работы.

Документ продолжает:

- [Anki Review Event Taxonomy](anki-review-event-taxonomy.md) — определяет, какое событие является `core`, `support`, `supplemental`, `none` или `deferred`;
- [Anki Review Reward Model](anki-review-reward-model.md) — определяет относительную стоимость допустимого `Review Episode`;
- [Anki XP Foundation](anki-xp-foundation.md) — задаёт общие инварианты домена Anki.

Главный вопрос третьего этапа:

> Как сделать манипуляцию Review XP невыгодной, не превратив защиту от abuse в наказание для обычного пользователя и не разрушив основную механику получения опыта?

Документ не проектирует:

- глобальные дневные diminishing returns;
- volume bonus;
- бонус за закрытие очереди;
- окончательную конвертацию Review Units в XP;
- стрик и `Momentum`;
- сетевой рейтинг;
- серверную верификацию;
- техническую схему хранения production-данных.

## 2. Центральный инвариант: anti-abuse не должен ломать награду

Защита от манипуляций является подчинённой частью экономики, а не её основной целью.

```text
Учебная ценность события
→ определяет награду.

Anti-abuse
→ предотвращает искусственное повторение или раздувание этой награды.
```

Неправильная модель:

```text
любое необычное поведение
→ подозрение
→ сильное снижение всего XP
```

Рекомендуемая модель:

```text
доказанное технически неучебное событие
→ 0 XP или no-op;

валидное учебное событие с сомнительным контекстом
→ базовая награда сохраняется;
→ ограничивается только дополнительный бонус;

неоднозначное событие
→ нейтральная награда или provisional state;
→ никаких обвинений и необратимых санкций.
```

### 2.1. Что защита не имеет права делать

Anti-abuse механизм не должен:

- уменьшать базовую награду только из-за одной слабой эвристики;
- штрафовать пользователя за обычное изменение настроек Anki;
- считать backlog доказательством мошенничества;
- считать filtered deck доказательством мошенничества;
- считать быстрый ответ доказательством автоматизации;
- лишать XP пользователя без FSRS;
- делать редкие, сложные или нестандартные карточки экономически бесполезными;
- скрыто менять награду без объяснения;
- отнимать уже подтверждённый постоянный XP из-за последующей эвристической переоценки;
- менять scheduler Anki;
- исправлять выбранные пользователем оценки;
- блокировать работу Anki;
- отправлять локальные данные наружу без отдельного согласия;
- создавать такой объём ограничений, при котором обычный Review перестаёт ощущаться вознаграждаемым.

### 2.2. Допустимое воздействие защиты

Контрмера может:

- не создать повторную транзакцию;
- отменить provisional reward вместе с Undo исходного review;
- классифицировать событие как `support`, `supplemental` или `none` по детерминированным данным;
- ограничить только `ContextCredit`;
- заменить неточный контекстный бонус нейтральным контекстом;
- установить cap на повторяемую последовательность;
- отложить подтверждение неоднозначного события;
- сохранить диагностическое объяснение;
- принять остаточный риск, если более строгая защита навредит честным сценариям.

## 3. Уточнение Reward Model

Во втором этапе использовалась формула, где `ResponseValidity` умножает весь core reward:

```text
CoreReviewUnits =
    ResponseValidity
  × (
        AttemptCredit
      + Pass × OutcomeCredit
      + Pass × ContextCredit
    )
```

Третий этап уточняет её для anti-abuse целей.

Одна эвристика времени или поведения не должна одновременно уменьшать:

- credit за честную попытку;
- credit за успешное воспроизведение;
- контекстный бонус.

Рекомендуемая логическая форма:

```text
BaselineCredit =
    AttemptCredit
  + Pass × OutcomeCredit

ContextBonus =
    Pass × ContextCredit

CoreReviewUnits =
    CoreEligibility × BaselineCredit
  + BonusEligibility × ContextBonus
```

Где:

```text
CoreEligibility  ∈ {0, 1}
BonusEligibility ∈ [0, 1]
```

### 3.1. CoreEligibility

`CoreEligibility = 0` допускается только при детерминированном основании:

- административная запись;
- дубликат уже награждённого source event;
- событие отменено через Undo;
- событие уже относится к другому core-эпизоду этой карточки в тот же Anki-день;
- событие маршрутизировано в Learn XP;
- технически невозможная или повреждённая запись, которую нельзя считать реальным ответом;
- доказанная повторная обработка одного и того же interaction.

Одна статистическая аномалия времени ответа не должна самостоятельно устанавливать `CoreEligibility = 0`.

### 3.2. BonusEligibility

`BonusEligibility` управляет только контекстным бонусом.

Примерная политика:

| Состояние | CoreEligibility | BonusEligibility |
|---|---:|---:|
| Обычное подтверждённое core-событие | `1` | `1.00` |
| Недостаточно FSRS-данных | `1` | нейтральный context |
| Умеренная просрочка | `1` | ограниченный bonus |
| Тяжёлый backlog | `1` | сильно ограниченный bonus |
| Подозрительно быстрое одиночное событие | `1` | `0–0.50` |
| Повторяющийся машинный burst | `1` или provisional | `0` |
| Forced due до естественного срока | не core | supplemental policy |
| Дубликат source event | `0` | `0` |
| Undo | `0` | `0` |
| Manual reschedule record | `0` | `0` |

### 3.3. Reward floor

Для валидного успешного core-события при отсутствии детерминированного основания для исключения:

```text
BaselineCredit = 0.25 + 0.65 = 0.90 Review Unit
```

Anti-abuse эвристики не должны уменьшать этот floor в локальном одиночном режиме.

Нейтральный контекст может вернуть обычное значение около:

```text
0.90 + 0.10 = 1.00 Review Unit
```

Это означает:

- сомнительный контекст лишает дополнительной выгоды;
- само реальное успешное повторение остаётся вознаграждаемым;
- экономика не превращается в систему постоянных штрафов.

Неуспешный core-эпизод сохраняет:

```text
AttemptCredit = 0.25 Review Unit
```

если событие не является административным, повторным или отменённым.

## 4. Модель угроз

Защищаемые активы:

1. корректность постоянного XP;
2. уникальность reward transactions;
3. относительную справедливость между честными способами обучения;
4. значение `Review Unit` как единицы реальной работы;
5. объяснимость начисления;
6. возможность честно изменить настройки Anki;
7. сохранность прошлого подтверждённого прогресса;
8. нормальный пользовательский опыт без навязчивого античита.

### 4.1. Уровни угрозы

#### Уровень A. Случайные технические аномалии

Пользователь не пытается получить лишний XP:

- повторная обработка cache;
- Sync с нескольких устройств;
- Undo;
- восстановление backup;
- импорт;
- изменение FSRS-параметров;
- изменение часового пояса;
- filtered deck;
- пересборка истории.

Это приоритетный класс защиты. Система обязана прежде всего не начислять двойную награду из-за собственных технических процессов.

#### Уровень B. Оптимизация через обычный интерфейс

Пользователь ищет выгодную игровую стратегию:

- специально откладывает карточки;
- меняет desired retention;
- делает карточки due вручную;
- повторяет будущие карточки;
- использует `Hard` вместо `Again`;
- создаёт много простых карточек;
- циклически меняет presets.

Это основной предмет текущей abuse-модели.

#### Уровень C. Автоматизация и изменение поведения Anki

- Auto Advance;
- макросы;
- custom scheduling;
- add-on, автоматически выбирающий оценки;
- массовые изменения расписания;
- программное создание дубликатов.

Наличие такой функции не является доказательством abuse. Оценивается конкретное событие и его игровой результат.

#### Уровень D. Полный контроль над локальной системой

- редактирование gamification DB;
- изменение Python-кода;
- подделка `revlog`;
- удаление ledger;
- модификация runtime;
- создание полностью синтетического профиля.

Для локального одиночного режима этот риск принимается.

Полностью защитить локальную программу от владельца устройства невозможно. Попытка сделать это создаст сложность и ложное ощущение безопасности, не решив фундаментальную проблему доверия.

## 5. Стратегия реакции

### 5.1. Сначала устранение стимула

Предпочтительная контрмера — экономика, в которой махинация не даёт значимого преимущества:

- `Hard`, `Good`, `Easy` имеют одинаковый direct outcome credit;
- `Easy` не повышает игровой memory gain;
- долгое ожидание не даёт положительный бонус;
- relearning имеет episode cap;
- backlog даёт только ограниченный delay credit;
- preview не создаёт постоянный XP;
- forced due не превращается в natural core review.

### 5.2. Затем детерминированная классификация

Если событие однозначно:

- duplicate;
- manual;
- undo;
- already rewarded;
- route to Learn;

применяется жёсткое правило без эвристики.

### 5.3. Эвристика влияет только на бонус

Если намерение нельзя доказать:

- базовая учебная награда сохраняется;
- challenge/memory bonus ограничивается;
- событие получает reason code;
- система не использует слово «читерство» в пользовательском интерфейсе.

### 5.4. Необратимые санкции не применяются

В локальной версии отсутствуют:

- бан;
- удаление уровня;
- отрицательный XP;
- скрытый shadow penalty;
- блокировка функций;
- автоматическое исправление Anki-состояния.

## 6. Уровни доказательности

| Уровень | Описание | Допустимая реакция |
|---|---|---|
| `deterministic` | Событие однозначно не создаёт новую учебную работу | `none`, no-op или reversal |
| `strong_context` | Несколько независимых сигналов указывают на повторяемую аномалию | убрать bonus, provisional state |
| `heuristic` | Один или несколько мягких сигналов | нейтральный context или ограничение bonus |
| `unknown` | Данных недостаточно | базовая или нейтральная награда |
| `accepted` | Нельзя защитить без непропорционального вреда | риск принимается |

### 6.1. Правило независимых сигналов

Сильное решение не должно основываться на одном признаке, если этот признак может возникнуть в честной работе.

Пример:

```text
очень быстрое время одного ответа
→ недостаточно для ограничения baseline;

очень быстрое время
+ длинный burst
+ машинно одинаковые интервалы
+ невозможность прослушать обязательное аудио
→ strong_context для отключения bonus.
```

## 7. Реестр abuse cases

## ABUSE-REV-001. Повторная обработка одного source event

**Категория:** технический duplicate  
**Возможная выгода:** многократное начисление полного XP  
**Обнаружение:** deterministic  
**Риск ложного срабатывания:** низкий

### Сценарий

Одна запись повторения обрабатывается повторно:

- после перезапуска;
- после sync;
- после rebuild;
- после миграции;
- после восстановления cache;
- после повторного импорта.

### Контрмера

Каждая reward transaction получает стабильный ключ:

```text
source_event_key
```

Логическая уникальность:

```text
UNIQUE(source_event_key, reward_channel)
```

### Реакция

```text
первое применение → transaction created;
повторное применение → no-op.
```

### Защита честного пользователя

Никаких штрафов и диагностических предупреждений. Повторная обработка является нормальной идемпотентной операцией.

### Остаточный риск

Нужно надёжно определить lineage профиля и imported history.

### Решение

`eliminate` для известных source events.

---

## ABUSE-REV-002. Несколько core-наград одной карточки за день

**Категория:** repeated event  
**Возможная выгода:** многократный core reward  
**Обнаружение:** deterministic при известном card lineage  
**Риск ложного срабатывания:** низкий

### Сценарий

Карточка многократно показывается через:

- filtered decks;
- Review Ahead;
- несколько сессий;
- ручное изменение срока;
- повторную сборку Custom Study.

### Контрмера

```text
UNIQUE(card_lineage, anki_day, core_review_channel)
```

### Реакция

- первый допустимый эпизод: `core`;
- relearning: `support`;
- ранняя дополнительная практика: `supplemental`;
- прочие повторы: `none`.

### Защита честного пользователя

Дополнительная практика остаётся видимой в статистике. Она просто не выдаёт второй полный core reward.

### Решение

`eliminate` для core-дубликатов.

---

## ABUSE-REV-003. Relearning loop

**Категория:** support farming  
**Возможная выгода:** большое число reward events из одного lapse  
**Обнаружение:** deterministic через parent episode  
**Риск ложного срабатывания:** низкий

### Сценарий

```text
Again → Good
Again → Good
Again → Good
```

или большое количество relearning steps.

### Контрмера

Все relearning events связываются через:

```text
parent_episode_id
```

Кандидатный cap:

```text
support одного lapse episode ≤ 0.12 Unit
```

### Реакция

Количество шагов не увеличивает общий cap.

### Защита честного пользователя

Человек получает небольшой support credit за реальную дополнительную работу, но не лишается `AttemptCredit` из-за ошибки.

### Решение

`mitigate` через episode cap.

---

## ABUSE-REV-004. Early review

**Категория:** scheduler optimization  
**Возможная выгода:** быстрый XP на лёгких будущих карточках  
**Обнаружение:** deterministic при известном natural due  
**Риск ложного срабатывания:** низкий

### Сценарий

Пользователь собирает будущие карточки в filtered deck или использует Review Ahead.

### Контрмера

```text
early review:
  class = supplemental;
  ChallengeCredit = 0;
  MemoryGainCredit = 0;
  отдельный card/day cap.
```

### Защита честного пользователя

Дополнительная практика перед экзаменом признаётся полезной и отображается, но не становится равноценной плановому retrieval event.

### Решение

`mitigate`.

---

## ABUSE-REV-005. Preview без rescheduling

**Категория:** repeatable no-state-change action  
**Возможная выгода:** бесконечный reward без изменения scheduler state  
**Обнаружение:** deterministic  
**Риск ложного срабатывания:** низкий

### Контрмера

```text
permanent Review XP = 0
```

Допускается отдельная неэкономическая метрика `Practice Credit` или счётчик дополнительной практики.

### Защита честного пользователя

Система не называет действие бесполезным и не скрывает выполненный объём. Оно просто не увеличивает постоянный уровень.

### Решение

`eliminate` для permanent XP.

---

## ABUSE-REV-006. Set Due Date, Reset и ручное reschedule

**Категория:** manual schedule manipulation  
**Возможная выгода:** превращение будущих карточек в core queue  
**Обнаружение:** deterministic при доступной истории операций  
**Риск ложного срабатывания:** средний при неполной истории

### Контрмера

Хранятся:

```text
natural_due_before
manual_operation_at
forced_due
```

Политика:

```text
карточка уже была naturally due
→ core допустим;

карточка была будущей и сделана due вручную
→ supplemental до natural due;

сама Manual-запись
→ 0 XP.
```

Reset не очищает lifetime reward state.

### Защита честного пользователя

Пользователь может свободно исправлять расписание. Ограничивается только дополнительная игровая выгода от искусственного срока.

### Решение

`mitigate`.

---

## ABUSE-REV-007. Массовое FSRS-rescheduling

**Категория:** administrative schedule rewrite  
**Возможная выгода:** искусственный backlog и ложный challenge bonus  
**Обнаружение:** deterministic для служебной операции; contextual для следующих reviews  
**Риск ложного срабатывания:** средний

### Контрмера

Разделяются:

```text
служебная reschedule-запись
→ 0 XP;

последующее реальное воспроизведение
→ BaselineCredit сохраняется.
```

Для следующего review:

```text
schedule_origin = bulk_fsrs_reschedule
```

И ограничиваются:

- bonus за административно созданную просрочку;
- bonus за скачок model state;
- будущий completion bonus, если очередь создана только операцией reschedule.

### Защита честного пользователя

Реальная работа после изменения параметров остаётся вознаграждаемой минимум базовым core reward.

### Решение

`mitigate`.

---

## ABUSE-REV-008. Desired retention cycling

**Категория:** settings optimization  
**Возможная выгода:** создание большого количества лёгких due events или искусственно сложной очереди  
**Обнаружение:** изменение настройки — deterministic; намерение — unknown  
**Риск ложного срабатывания:** высокий

### Сценарии

```text
0.90 → 0.99
→ больше лёгких reviews;

0.90 → низкое значение
→ длинные интервалы;
→ последующий возврат и попытка получить challenge bonus.
```

### Контрмера: Reward Reference Policy

Разделяются:

```text
Scheduler Policy
— когда Anki показывает карточку;

Reward Reference Policy
— относительно какой нормы рассчитывается bonus.
```

Хранятся:

```text
reference_retention
reference_parameter_version
effective_from
stabilization_window
```

### Реакция

- базовый reward за реальные reviews сохраняется;
- смена настройки не создаёт мгновенный новый bonus pool;
- challenge bonus использует стабильную reference policy;
- переход reference policy происходит постепенно или после stabilization window.

### Защита честного пользователя

Нет штрафа за выбор нужной retention. Пользователь может изменить scheduler немедленно. Ограничивается только игровая выгода от циклического переключения.

### Решение

`mitigate`, требует симуляции.

---

## ABUSE-REV-009. Намеренный backlog

**Категория:** delay farming  
**Возможная выгода:** максимальный ChallengeCredit на большом числе карточек  
**Обнаружение:** факт задержки — deterministic; намерение — unknown  
**Риск ложного срабатывания:** высокий

### Контрмера

Reward Model использует:

```text
R_actual
R_natural_due
DelayCredit cap
```

```text
естественная сложность
→ полный ChallengeCredit;

сложность, созданная задержкой
→ ограниченный DelayCredit.
```

### Защита честного пользователя

Backlog может возникнуть из-за болезни, работы, поездки или выгорания.

Поэтому сохраняются:

- AttemptCredit;
- OutcomeCredit;
- естественный контекст;
- небольшой recovery/delay credit.

UI объясняет ограничение бонуса, но не обвиняет пользователя.

### Решение

`mitigate` без санкций.

## ABUSE-REV-010. Использование Hard вместо Again

**Категория:** answer gaming  
**Возможная выгода:** получить успешный outcome при фактическом забывании  
**Обнаружение:** намерение нельзя определить надёжно  
**Риск ложного срабатывания:** высокий

### Контрмера

- `Hard`, `Good`, `Easy` имеют одинаковый direct OutcomeCredit;
- фактический `Easy` не увеличивает игровой memory gain;
- последующая история используется для калибровки модели, но не для ретроактивного обвинения;
- система объясняет правильное назначение кнопок, не меняя scheduler.

### Защита честного пользователя

Никакого автоматического штрафа за `Hard`. Пользователь мог действительно вспомнить ответ с трудом.

### Решение

`reduce incentive`; остаточный риск принимается.

## ABUSE-REV-011. Ложный Good или Easy

**Обнаружение:** надёжно невозможно  
**Риск ложного срабатывания:** неприемлемый

Экономика не даёт прямой выгоды конкретной успешной кнопке.

Обман:

- почти не повышает игровой reward;
- ухудшает собственное расписание;
- может привести к будущим lapse;
- не создаёт отдельный memory gain bonus через `Easy`.

### Защита честного пользователя

Система не пытается угадывать содержимое мыслей и не переписывает оценки.

### Решение

`accept` после устранения основного стимула.

## ABUSE-REV-012. Сверхбыстрые ответы и макросы

**Категория:** automation or spam  
**Возможная выгода:** высокий XP/hour без осмысленного просмотра  
**Обнаружение:** heuristic/strong_context  
**Риск ложного срабатывания:** высокий для отдельных быстрых ответов

### Сигналы

- время значительно ниже персональной нормы;
- длинный burst;
- почти одинаковые интервалы;
- невозможная скорость для обязательного аудио;
- машинная периодичность;
- экстремальная доля одинаковых оценок;
- отсутствие естественных пауз между карточками.

### Контрмера

Один быстрый ответ:

```text
BaselineCredit сохраняется;
ContextBonus может быть ограничен.
```

Сильный повторяющийся burst:

```text
BonusEligibility = 0;
reward может остаться provisional до окончания сессии.
```

В локальном одиночном режиме baseline не обнуляется только из-за поведенческой эвристики.

### Защита честного пользователя

- быстрые простые карточки остаются выгодными;
- Auto Advance не считается нарушением сам по себе;
- аудио и тип карточки входят в expected time;
- защита реагирует на последовательность, а не на один эпизод.

### Решение

`mitigate bonus`, не ломая baseline.

## ABUSE-REV-014. Изменение системного времени или часового пояса

**Категория:** clock manipulation  
**Возможная выгода:** искусственные due dates, дополнительные Anki-дни, стрик  
**Обнаружение:** heuristic  
**Риск ложного срабатывания:** высокий

### Сигналы

```text
wall clock
monotonic session time
revlog order
previous observed UTC
timezone changes
Anki day boundary
```

### Контрмера

При существенном скачке:

```text
clock_confidence = low
```

- time-derived bonus отключается;
- overdue bonus ограничивается;
- day/streak finalization откладывается до стабилизации;
- прошлый XP не конфискуется.

### Защита честного пользователя

Путешествия, переход часового пояса и исправление неверных часов не приводят к потере основной награды.

### Решение

`mitigate bonus`; абсолютная локальная защита невозможна.

## ABUSE-REV-015. Exact duplicate cards

**Категория:** content duplication  
**Возможная выгода:** множество одинаковых лёгких core events  
**Обнаружение:** deterministic для exact duplicates; heuristic для semantic duplicates  
**Риск ложного срабатывания:** средний

### Контрмера

Нормализованный fingerprint:

```text
normalized question
+ normalized answer
+ card template
+ media identity
+ direction
```

Классы:

```text
exact_duplicate
probable_duplicate
sibling_related
independent
```

### Реакция

- exact duplicate одного направления: общий reward group;
- probable duplicate: только ограничение bonus или diagnostic;
- sibling: отдельный core event, но возможна метка `sibling_exposed`;
- semantic similarity никогда не обнуляет baseline автоматически.

### Защита честного пользователя

Recognition, production, аудио и cloze могут проверять разные retrieval prompts. Они не считаются дубликатами только из-за общей заметки.

### Решение

`mitigate exact`; `defer semantic`.

## ABUSE-REV-016. Sibling exposure

**Категория:** answer leakage  
**Возможная выгода:** ответ подсказан ранее показанным sibling  
**Обнаружение:** deterministic для порядка показа; влияние на знание — unknown  
**Риск ложного срабатывания:** высокий

### Контрмера

```text
class = core
context tag = sibling_exposed
```

Возможное ограничение применяется только к `ChallengeCredit`, а не к baseline.

### Защита честного пользователя

Разные siblings могут проверять разные факты или направления. Автоматическое обнуление reward запрещено.

### Решение

`defer exact modifier` до симуляции.

## ABUSE-REV-017. Sync, import, backup и replay истории

**Категория:** provenance ambiguity  
**Возможная выгода:** повторное начисление старой истории  
**Обнаружение:** deterministic/unknown в зависимости от lineage  
**Риск ложного срабатывания:** средний

### Требуемые идентификаторы

```text
source_event_key
review_episode_id
profile_lineage_id
collection_lineage_id
provenance
first_seen_at
reward_rule_version
```

### Контрмера

- известный source event: идемпотентный no-op;
- native historical event: отдельная historical policy;
- imported ambiguous event: neutral/deferred credit;
- восстановление cache не создаёт новый ledger entry;
- backup rollback не удаляет подтверждённую транзакцию без reconciliation policy.

### Защита честного пользователя

Обычный Sync и восстановление коллекции не должны приводить ни к двойному XP, ни к потере уже подтверждённого прогресса.

### Решение

`mitigate`, отдельная lineage specification потребуется позже.

## ABUSE-REV-018. Undo

**Категория:** reversed interaction  
**Возможная выгода:** получить XP, отменить review и повторить  
**Обнаружение:** deterministic  
**Риск ложного срабатывания:** низкий

### Контрмера

Reward проходит состояния:

```text
provisional
→ confirmed
```

Если исходный review отменён:

```text
provisional reward removed
или
confirmed transaction reversed
в рамках того же interaction.
```

Это не считается конфискацией постоянного XP: пользователь отменил само действие, на котором награда была основана.

### Защита честного пользователя

Undo продолжает работать штатно. Пользователь не получает отдельный штраф.

### Решение

`eliminate`.

**Категория:** scheduler modification  
**Возможная выгода:** искусственные состояния и сроки  
**Обнаружение:** факт включения — deterministic; конкретное влияние — contextual  
**Риск ложного срабатывания:** высокий

### Контрмера

Само наличие Custom Scheduling не снижает reward.

Snapshot эпизода сохраняет:

```text
custom_scheduling_enabled
scheduler_version
schedule_origin
natural_due
actual_due
reward_reference_version
```

Если естественный контекст нельзя восстановить:

- baseline сохраняется;
- используется neutral context;
- bonus не строится на недоверенном schedule state.

### Защита честного пользователя

Пользователь может применять полезный custom scheduler без автоматического штрафа.

### Решение

`mitigate bonus`; не запрещать функцию.

## ABUSE-REV-020. Прямое изменение reward ledger или кода

**Категория:** local owner control  
**Возможная выгода:** произвольный XP  
**Обнаружение:** ненадёжно против мотивированного владельца  
**Риск ложного срабатывания:** не применимо

### Возможные меры целостности

- hash chain;
- checksum;
- append-oriented ledger;
- reconciliation;
- backup;
- schema constraints.

Они защищают от случайного повреждения и простого редактирования, но не от пользователя, который может изменить и данные, и проверяющий код.

### Решение

```text
Local Progress
→ риск принимается;

будущий Competitive Verified Progress
→ требует доверенной внешней стороны.
```

`accept` для локального режима.

## ABUSE-REV-021. Изменение содержимого карточки между reviews

**Категория:** content drift  
**Возможная выгода:** превратить сложную карточку в тривиальную, сохранив выгодное FSRS-состояние  
**Обнаружение:** deterministic при наличии content fingerprint history  
**Риск ложного срабатывания:** средний

### Сценарий

Карточка накопила низкую retrievability или высокую difficulty, после чего вопрос заменяется очевидным ответом без reset.

### Контрмера

Snapshot содержит:

```text
content_fingerprint
content_revision
semantic_change_class
```

Существенное изменение:

- не отменяет право на baseline;
- временно сбрасывает или нейтрализует ContextCredit;
- может открыть Create XP за полезное исправление;
- не сохраняет искусственно высокий challenge bonus от старого содержания.

### Защита честного пользователя

Исправление плохой карточки является желательным действием. Пользователь не штрафуется и может получить Create XP позже. Ограничивается только несоответствующий старый memory context.

### Решение

`mitigate bonus`.

## ABUSE-REV-022. Preset hopping между колодами

**Категория:** configuration cycling  
**Возможная выгода:** выбирать наиболее выгодную комбинацию параметров непосредственно перед review  
**Обнаружение:** deterministic для смены preset; намерение — unknown  
**Риск ложного срабатывания:** высокий

### Контрмера

Reward context использует versioned snapshot и stabilization window.

- baseline сохраняется;
- резкая смена preset не создаёт новый ContextCredit;
- естественная работа с несколькими presets остаётся разрешённой;
- сравнение происходит внутри устойчивой reference policy.

### Решение

`mitigate bonus`, требует симуляции.

## 9. Защита от накопления ограничений

Несколько контрмер не должны независимо штрафовать одно событие за одну и ту же причину.

Неправильно:

```text
backlog penalty
× low retrievability penalty
× schedule-change penalty
× low-confidence penalty
× time penalty
```

Такой pipeline способен уничтожить нормальную награду.

### 9.1. Причинная дедупликация ограничений

Каждое ограничение получает `reason_group`:

```text
provenance
schedule_manipulation
delay
response_pattern
content_identity
duplicate_processing
model_confidence
```

Для одной причины применяется одно наиболее подходящее решение.

Пример:

```text
bulk FSRS reschedule
→ schedule_manipulation reason;

низкая R, возникшая из-за этого же reschedule
→ не создаёт второй независимый penalty.
```

### 9.2. Anti-abuse impact cap

Для эвристических ограничений:

```text
BaselineCredit не уменьшается;
ContextBonus может быть уменьшен не ниже 0;
итог не становится отрицательным;
прошлый confirmed XP не пересчитывается автоматически.
```

### 9.3. Один episode — один explainable outcome

Пользователь видит не пять штрафов, а одно итоговое объяснение:

```text
Базовая награда: 0.90
Контекстный бонус: 0.00
Причина: повторение произошло после ручного изменения срока.
Итого: 0.90 Review Unit
```

или:

```text
Событие уже было учтено.
Дополнительный XP не начислен.
```

## 10. Пользовательские формулировки

Нельзя использовать без надёжного доказательства:

- «читерство»;
- «мошенничество»;
- «нарушение»;
- «подозрительный пользователь»;
- «штраф».

Рекомендуемые формулировки:

| Ситуация | Объяснение |
|---|---|
| Backlog | «Базовая награда сохранена; дополнительный бонус ограничен из-за длительной просрочки» |
| Forced due | «Карточка повторена раньше естественного срока; учтена как дополнительная практика» |
| Duplicate | «Это событие уже было учтено» |
| Undo | «Начисление отменено вместе с повторением» |
| Low model confidence | «Использована нейтральная оценка сложности» |
| Fast burst | «Контекстный бонус не применён из-за недостатка достоверных данных о попытке» |
| Bulk reschedule | «Повторение учтено; бонус расписания ограничен после массового пересчёта» |
| Preview | «Дополнительная практика записана без постоянного XP» |

## 12. Проверочные примеры

### Пример A. Обычный быстрый ответ

```text
Карточка короткая и знакомая.
Ответ: 1.2 секунды.
Предыдущая медиана: 1.5 секунды.
```

Результат:

```text
BaselineCredit: сохранён
ContextBonus: полный
Причина: время соответствует карточке
```

Защита не считает любой быстрый ответ abuse.

### Пример B. Одиночный экстремально быстрый ответ

```text
Ответ: 0.15 секунды.
Остальная сессия нормальная.
```

Результат:

```text
BaselineCredit: сохранён
ContextBonus: ограничен или neutral
Никаких санкций для следующих карточек
```

### Пример C. Машинный burst

```text
300 карточек.
Время каждой: 0.10–0.11 секунды.
Одинаковая оценка.
Карточки содержат обязательное аудио.
```

Результат:

```text
BaselineCredit: local policy сохраняет или временно делает provisional
ContextBonus: 0
Volume bonus: недоступен
Сессия помечается для diagnostic breakdown
```

Точная политика baseline должна проверяться симуляцией, чтобы не создать выгодный automation path.

### Пример D. Честное возвращение после болезни

```text
Просрочка: 21 день.
R_actual: 0.35.
Пользователь успешно вспомнил карточку.
```

Результат:

```text
BaselineCredit: сохранён
Natural challenge: сохранён
DelayCredit: ограничен
Итого: выше обычного review, но не максимальный фарм-бонус
```

### Пример E. Изменение desired retention ради качества обучения

```text
Пользователь меняет 0.90 → 0.95.
Причина: хочет меньше забываний.
```

Результат:

```text
Scheduler работает по 0.95 немедленно
BaselineCredit всех реальных reviews сохраняется
Reward reference стабилизируется отдельно
Никаких штрафов за настройку
```

### Пример F. Циклическая смена retention

```text
0.90 → 0.99 → review queue → 0.90
```

Результат:

```text
BaselineCredit: сохраняется
ContextBonus: не создаётся из мгновенной смены
Настройка не блокируется
```

### Пример G. Массовый reschedule после оптимизации FSRS

```text
500 карточек стали due.
Пользователь честно повторил 80.
```

Результат:

```text
служебные записи: 0 XP
80 реальных reviews: BaselineCredit сохранён
administrative challenge bonus: ограничен
```

### Пример H. Undo после ошибочной кнопки

```text
Пользователь нажал Easy вместо Again.
Сразу сделал Undo.
Повторил карточку честно.
```

Результат:

```text
первая provisional transaction отменена
вторая реальная попытка классифицирована заново
двойного XP нет
штрафа нет
```

### Пример I. Filtered deck с due-карточками

```text
Карточки уже naturally due.
Rescheduling включён.
```

Результат:

```text
обычный core reward
filtered deck сам по себе не снижает XP
```

### Пример J. Filtered preview

```text
Rescheduling выключен.
Карточки повторяются несколько раз.
```

Результат:

```text
practice statistics: учитываются
permanent Review XP: 0
```

### Пример K. Улучшение проблемной карточки

```text
Карточка была сложной.
Пользователь исправил двусмысленный вопрос.
```

Результат:

```text
следующий review получает BaselineCredit
старый ContextCredit нейтрализован из-за content revision
полезное исправление может получить Create XP отдельно
```

Пользователь не наказывается за улучшение материала.

### Пример L. Exact duplicate после импорта

```text
Импортирована копия существующей карточки.
```

Результат:

```text
exact duplicate group обнаружен
второй экземпляр не создаёт независимый полный reward stream
исходные подтверждённые награды не удаляются
```

### Пример M. Путешествие в другой часовой пояс

```text
Часовой пояс изменился на +7 часов.
Пользователь продолжил обычные reviews.
```

Результат:

```text
BaselineCredit: сохранён
clock-derived bonus: neutral/limited
стрик финализируется после стабилизации
прошлый XP не изменяется
```

## 13. Acceptance criteria для будущей симуляции

Следующие значения являются целевыми гипотезами, а не окончательными SLA.

### 13.1. Сохранение честной экономики

Для обычных честных сценариев:

```text
не менее 99% core events сохраняют BaselineCredit;
не менее 95% получают ожидаемый neutral/full ContextCredit;
baseline removal из-за одной эвристики = 0 случаев;
```

Для честных edge workflows:

- filtered deck с naturally due cards;
- Auto Advance;
- длинное аудио;
- поездка;
- backlog после перерыва;
- изменение desired retention;
- custom scheduler;

средний итоговый reward не должен необъяснимо снижаться более чем на `5%` относительно сопоставимой учебной работы, кроме заранее принятых supplemental-классов.

### 13.2. Ограничение exploit-выгоды

Симулятор должен проверить:

- repeated processing не даёт больше `0%` дополнительного XP;
- relearning loop не превышает episode cap;
- preview не создаёт permanent XP;
- forced due не превосходит natural core strategy;
- intentional backlog не превосходит регулярное обучение по ожидаемому XP на единицу реальной работы;
- desired retention cycling не создаёт устойчивого bonus advantage;
- систематический `Easy` не увеличивает игровой reward;
- удерживание карточки открытой не увеличивает reward;
- duplicate cards не масштабируют XP линейно;
- несколько защитных правил не складываются в чрезмерный штраф.

### 13.3. Explainability

Для каждого ограниченного события доступны:

```text
base reward
context reward
support reward
classification
reason code
rule version
```

### 13.4. Reversibility

- provisional reward можно подтвердить или отменить;
- Undo корректно отменяет только связанную транзакцию;
- изменение эвристики не пересчитывает прошлый confirmed XP автоматически;
- новую rule version можно симулировать отдельно.

## 14. Параметры, которые остаются гипотезами

- точный reward floor;
- сохранение baseline при доказанном машинном burst в local mode;
- stabilization window для retention и preset changes;
- размер DelayCredit;
- exact duplicate reward grouping;
- влияние sibling exposure;
- expected response time model;
- границы clock anomaly;
- политика historical/imported XP;
- необходимость отдельного `Practice Credit`;
- длительность provisional window;
- допустимый impact cap для нескольких reason groups.

## 15. Решения третьего этапа

Предварительно приняты:

1. Anti-abuse не является основанием ослаблять нормальную механику получения XP.
2. Детерминированно неучебные события исключаются; неоднозначные сохраняют baseline.
3. Эвристики прежде всего управляют `ContextBonus`, а не `BaselineCredit`.
4. Одна слабая эвристика не может обнулить core reward.
5. `Hard`, `Good`, `Easy` не имеют разных прямых наград.
6. Backlog не считается доказательством abuse.
7. Настройки Anki не блокируются и не штрафуются сами по себе.
8. Reward Reference Policy отделяется от Scheduler Policy.
9. Повторная обработка должна быть идемпотентной.
10. Relearning ограничивается на уровне episode.
11. Manual и administrative records не получают XP.
12. Preview без rescheduling не создаёт permanent XP.
13. Forced due не становится natural core event.
14. Undo отменяет связанную provisional transaction без штрафа.
15. Несколько контрмер не штрафуют одно событие повторно за одну причину.
16. Semantic similarity не может автоматически удалить baseline.
17. Полный контроль владельца над локальной системой является принимаемым риском.
18. Любая более строгая конкурентная верификация должна быть отдельным режимом.
19. Каждое ограничение должно быть объяснимым.
20. Контрмеры обязаны пройти симуляцию честных edge cases до реализации.

## 16. Следующий этап

Следующий документ:

```text
anki-review-session-and-day.md
```

Он должен определить:

- что является Review Session;
- как суммируются `Review Units`;
- как учитываются несколько сессий за Anki-день;
- что считается уникальным объёмом работы;
- нужен ли volume bonus;
- как избежать двойного anti-grind после episode-level safeguards;
- как определяется минимальный и полноценный Review-день;
- как учитываются backlog, support и supplemental work;
- как Review XP участвует в будущем стрике;
- как anti-abuse impact caps взаимодействуют с дневной агрегацией.

Ключевой вопрос следующего этапа:

> Как превратить множество корректно оценённых Review Episodes в справедливую награду за сессию и день, не создавая второй слой наказаний поверх уже применённых safeguards?

## Stage 5B.C matched longitudinal correction

Retention cycling and intentional backlog are now evaluated on persistent
matched cohorts rather than independent daily samples. A comparison must share
the initial cohort digest and common latent recall stream; only the retention
timeline or delay window may differ. Reported exploit advantage subtracts the
CoreBaseline attached to additional legitimate scheduled reviews before the
remainder is normalized by the control total. Thus a policy is not labeled an
exploit merely because it schedules more real work.

The 90-day cycling and intentional-backlog gates are `<= 3%` unexplained
advantage. Missing matched evidence is `UNSUPPORTED`, never numeric zero.
