# Gamification Progression Foundation

Статус: **DRAFT v0.2**  
Дата: **2026-07-16**  
Область: **независимый продуктовый концепт, пока не являющийся частью Anki Study Report**

## 1. Назначение документа

Этот документ фиксирует общий фундамент будущей системы игрофикации личного развития:

- уровни;
- постоянный опыт (`XP`);
- градацию долгосрочного прогресса;
- стрик регулярности;
- временный темп (`Momentum`);
- запланированный отдых;
- защищённые и незащищённые пропуски;
- санкции;
- восстановление после перерыва;
- общие правила против grind и чрезмерного наказания.

Система проектируется как самостоятельная модель, которая в будущем может охватывать Anki, изучение языков, программирование, физическую активность, проекты и другие области.

Первым подробно прорабатываемым доменом является Anki. Его источники XP и доменные антифарм-правила описаны отдельно в [anki-xp-foundation.md](anki-xp-foundation.md).

Документ не является технической спецификацией реализации. Числовые значения и формулы задают первую проверяемую модель и должны пройти симуляцию до признания окончательными.

## 3. Центральная модель

Система разделяет три вида прогресса:

```text
Уровень  — вся накопленная история подтверждённых полезных усилий.
Стрик    — непрерывность выполнения выбранных обязательств.
Momentum — текущая форма и временное преимущество за регулярность.
```

Это разделение является базовым инвариантом.

Один показатель не должен одновременно отвечать за долгосрочный путь, непрерывность и текущее состояние. Иначе пропуск либо почти ничего не значит, либо уничтожает месяцы и годы работы.

### 3.1. Уровень

Уровень показывает суммарный подтверждённый прогресс пользователя.

Свойства:

- зависит от общего заработанного XP;
- никогда не уменьшается;
- не сбрасывается после пропуска;
- не является прямой оценкой интеллекта, знаний или профессионального ранга;
- растёт всё медленнее по мере продвижения;
- должен отражать месяцы и годы полезной деятельности;
- не требует постоянного увеличения ежедневной нагрузки.

Высокий уровень означает не обязанность работать по десять часов в день, а значительный накопленный путь.

### 3.2. Стрик

Стрик показывает непрерывную последовательность выполненных обязательных дней.

Свойства:

- может завершиться;
- имеет текущую длину и личный рекорд;
- учитывает выбранное расписание;
- не ломается из-за заранее предусмотренного отдыха;
- может быть сохранён ограниченной защитой;
- не даёт бесконечно растущий множитель XP;
- служит прежде всего подтверждением устойчивости и источником вех.

### 3.3. Momentum

`Momentum` показывает текущую регулярность и управляет временным XP-бонусом.

Свойства:

- диапазон `0–100`;
- растёт за выполненные обязательства;
- не растёт от переработки сверх нормы;
- немного падает при защищённом пропуске;
- сильнее падает при обычном пропуске;
- быстро восстанавливается после возвращения;
- не влияет на уже заработанный XP;
- не может стать отрицательным.

## 4. Основные продуктовые принципы

### 4.1. Постоянный прогресс не конфискуется

Система не должна:

- отнимать заработанный XP;
- понижать уровень;
- создавать отрицательный баланс опыта;
- удалять уже открытые постоянные награды;
- аннулировать многомесячный путь из-за короткого перерыва.

Санкции воздействуют на временные преимущества и текущую непрерывность, но не переписывают прошлое.

### 4.2. Высокий уровень требует времени, а не бесконечной дневной нагрузки

Неправильная модель:

```text
Уровень 5:   30 минут в день.
Уровень 50:  5 часов в день.
Уровень 100: 12 часов в день.
```

Целевая модель:

```text
Уровень 5:   несколько продуктивных дней.
Уровень 50:  длительный устойчивый период.
Уровень 100: годы регулярной работы.
```

Рост сложности обеспечивается увеличением суммарного XP между уровнями, а не обязательным повышением ежедневной нормы.

### 4.3. Работа важнее присутствия

XP не начисляется только за:

- открытое приложение;
- время на экране;
- запущенный таймер;
- символическую активность;
- бессодержательные повторяемые действия.

Каждый домен обязан определить собственные проверяемые события полезной работы.

### 4.4. Возвращение всегда выгоднее отказа

Пропуск должен ощущаться как потеря текущего преимущества, но не должен делать возвращение бессмысленным.

> Пользователь теряет временный темп, но не право продолжить путь.

### 4.5. Отдых не является ошибкой

Запланированный отдых является частью нормального режима. Он не требует оправдания, не тратит защиту и не считается нарушением.

### 4.6. Стрик не должен провоцировать фиктивную активность

День засчитывается только после содержательного минимального порога. Один клик, одна секунда активности или дешёвая повторяемая операция не должны сохранять серию.

## 5. Базовая кривая уровней

### 5.1. Первый цикл

Первый крупный цикл проектируется на **100 уровней**.

Это не обязательно вечный максимум. Позже могут появиться:

- уровни сверх 100;
- дополнительные циклы;
- престиж;
- доменные ранги.

Эти механики пока не проектируются.

### 5.2. Требование до следующего уровня

Первая формула:

```text
XPnext(L) = roundTo5(100 + 25 × L^1.2)
```

Где:

- `L` — текущий уровень, начиная с `1`;
- `roundTo5` — округление до ближайшего числа, кратного пяти.

Примеры:

| Переход | XP до следующего уровня |
|---|---:|
| `1 → 2` | `125` |
| `5 → 6` | `270` |
| `10 → 11` | `495` |
| `20 → 21` | `1 010` |
| `25 → 26` | `1 290` |
| `40 → 41` | `2 190` |
| `50 → 51` | `2 835` |
| `75 → 76` | `4 545` |
| `100 → 101` | `6 380` |

Для достижения уровня 100 с первого уровня потребуется приблизительно:

```text
292 225 XP
```

### 5.3. Расчётный продуктивный день

Для первичной калибровки используется условная единица:

```text
200 Base XP = один полноценный продуктивный день
```

Это не означает конкретное количество минут, карточек или задач.

После появления Anki XP Foundation значение `200 XP` рассматривается только как целевой масштаб для симуляции. Оно не должно определять стоимость отдельного review, новой карточки или редактирования до проверки реальных распределений.

При постоянном среднем доходе:

| Средний XP в день | Ориентировочное время до уровня 100 |
|---:|---:|
| `150` | около `5,3 года` |
| `200` | около `4 лет` |
| `300` | около `2,7 года` |

### 5.4. Почему используется степенная кривая

Экспоненциальная формула быстро делает высокие уровни недостижимыми или требует постоянной инфляции наград.

Умеренная степенная кривая:

- даёт быстрый старт;
- постепенно увеличивает интервал между уровнями;
- остаётся прогнозируемой;
- не требует ежегодно умножать все награды;
- позволяет сохранять ценность одной единицы XP.

## 6. Общая экономика XP

### 6.1. Base XP

`Base XP` начисляется за подтверждённое полезное усилие.

Правила:

- одинаковое по ценности усилие должно давать сопоставимый XP независимо от уровня;
- высокий уровень не снижает номинальную ценность действия;
- увеличение времени до следующего уровня обеспечивается кривой уровня;
- каждый домен определяет собственные источники XP;
- повторяемые дешёвые действия получают доменную защиту от фарма;
- время само по себе не считается результатом.

### 6.2. Доменный расчёт применяется первым

После проработки Anki принято уточнение:

> Общий дневной diminishing returns нельзя применять непосредственно к необработанным событиям.

Правильный порядок:

```text
source event
→ domain classification
→ validation
→ domain reward formula
→ same-item and lifetime safeguards
→ domain-specific diminishing returns
→ aggregation into Base XP
→ global daily diminishing returns
→ Momentum bonus
→ final XP
```

Это предотвращает ситуацию, когда дешёвые и дорогие события ошибочно считаются равными до проверки их полезности.

### 6.3. Глобальное мягкое снижение дневной эффективности

Первая модель общего дневного envelope:

```text
первые 200 domain-validated XP:       100% эффективности;
следующие 150 domain-validated XP:     60% эффективности;
всё после 350 domain-validated XP:     25% эффективности.
```

Формула:

```text
BaseXP =
    min(validatedXP, 200)
  + 0.60 × min(max(validatedXP − 200, 0), 150)
  + 0.25 × max(validatedXP − 350, 0)
```

Примеры:

| Domain-validated XP | Итоговый Base XP |
|---:|---:|
| `100` | `100` |
| `200` | `200` |
| `300` | `260` |
| `350` | `290` |
| `500` | `327,5` |
| `1 000` | `452,5` |

Округление пока не зафиксировано.

### 6.4. Статус глобального diminishing returns

Эта формула является гипотезой, а не принятым балансом.

Anki XP simulation должна проверить:

- не применяется ли слишком много ограничений одновременно;
- не обесцениваются ли длинные честные сессии;
- не возникает ли двойной штраф после domain-specific caps;
- достаточно ли защищена система без глобального третьего диапазона;
- соответствует ли итоговый дневной XP кривой уровней.

Глобальный envelope может быть изменён или заменён после симуляции.

## 7. Обязательства и расписание

### 7.1. Стрик обязательств

Основная серия строится вокруг выполнения выбранного режима, а не обязательного занятия каждый календарный день.

Концептуальные варианты:

```text
7 обязательных дней из 7;
6 обязательных дней из 7;
5 обязательных дней из 7;
собственное недельное расписание.
```

Пример:

```text
Понедельник–суббота: обязательные дни.
Воскресенье: запланированный отдых.
```

Воскресенье:

- не увеличивает стрик;
- не прерывает стрик;
- не требует защиты;
- не даёт streak-бонус;
- является нормальной частью режима.

### 7.2. Статусы дня

| Статус | Значение |
|---|---|
| `complete` | Достигнут минимальный полезный порог; стрик продвигается |
| `partial` | Полезная активность была, XP начисляется, но порог не достигнут |
| `scheduled_rest` | Заранее предусмотренный отдых; стрик не меняется |
| `protected_skip` | Использована защита; стрик сохраняется без продвижения |
| `missed` | Обязательство не выполнено; текущий стрик завершается |
| `repaired` | Однодневный разрыв восстановлен по специальным правилам |

### 7.3. Минимум для зачёта дня

Предварительная калибровка:

```text
50 Base XP  = минимальный порог зачёта дня;
200 Base XP = полноценный продуктивный день.
```

После появления Anki XP Foundation эти числа считаются только общей гипотезой.

Для каждого домена потребуется проверить:

- какие события могут участвовать в зачёте дня;
- какая доля дешёвых событий допустима;
- может ли один канал полностью закрыть обязательство;
- не позволяет ли минимальный порог фармить стрик;
- не слишком ли высок порог для тяжёлого дня.

Для Anki в порог потенциально входят Review XP, подтверждённые этапы Learn XP и ограниченная доля Create XP.

## 8. Momentum

### 8.1. Диапазон

```text
0 ≤ Momentum ≤ 100
```

### 8.2. Изменение

Первая модель:

| Событие | Изменение |
|---|---:|
| Выполнен обязательный день | `+10` |
| Первый выполненный день после незащищённого пропуска | `+15` |
| Запланированный отдых | `0` |
| Защищённый пропуск | `−5` |
| Первый незащищённый пропуск | `−25` |
| Второй пропуск подряд | ещё `−25` |
| Последующие пропуски | снижение до `0`, но не ниже |

Momentum не увеличивается сильнее только потому, что пользователь заработал значительно больше дневной нормы.

### 8.3. XP-бонус

Первая формула:

```text
MomentumBonus = min(BaseXP, 200) × Momentum × 0.002
```

| Momentum | Эффективный бонус к первым 200 Base XP |
|---:|---:|
| `0` | `0%` |
| `25` | `5%` |
| `50` | `10%` |
| `75` | `15%` |
| `100` | `20%` |

Максимальный ежедневный бонус:

```text
200 × 20% = 40 XP
```

Пример:

```text
Base XP:       180
Momentum:       70
MomentumBonus:  25,2
Total XP:      205,2
```

Округление пока не принято.

### 8.4. Назначение ограничения

Ограничение должно:

- делать регулярность заметной;
- не превращать длинный стрик в обязательное условие нормального прогресса;
- не создавать огромный разрыв между новым и старым пользователем;
- снижать страх потери серии;
- не позволять стрику разгонять инфляцию XP.

## 9. Стрик

### 9.1. Что увеличивает стрик

Стрик увеличивается на один только тогда, когда обязательный день получает статус `complete`.

### 9.2. Что не увеличивает стрик

Стрик не увеличивают:

- `partial`;
- `scheduled_rest`;
- `protected_skip`;
- день без обязательства;
- восстановленный пропуск сам по себе;
- дополнительная работа сверх дневного порога.

### 9.3. Что завершает стрик

Текущий стрик завершается, если обязательный день получает статус `missed` и не был защищён или восстановлен допустимым способом.

История должна сохранять:

- завершённый стрик;
- дату начала и завершения;
- личный рекорд;
- число защищённых дней;
- число восстановлений.

### 9.4. Вехи

Предварительные вехи:

```text
3, 7, 14, 30, 50, 100, 180, 365 дней.
```

Вехи в основном должны давать:

- достижения;
- титулы;
- косметические награды;
- ограниченную защиту;
- визуальное развитие профиля.

XP-награды должны быть небольшими и фиксированными.

Предварительный пример:

| Стрик | XP | Дополнительная награда |
|---:|---:|---|
| `7` | `50` | одна защита |
| `30` | `150` | достижение |
| `100` | `300` | титул |
| `365` | `1 000` | уникальная косметика |

Значения не считаются окончательными.

## 10. Запланированный отдых

`scheduled_rest` является полноценным состоянием, а не разновидностью пропуска.

Правила:

- задаётся расписанием заранее;
- не расходует защиту;
- не снижает Momentum;
- не увеличивает стрик;
- не даёт XP сам по себе;
- не создаёт необходимость что-либо отрабатывать;
- не считается нарушением режима.

Позже потребуется определить ограничения на изменение расписания, чтобы пропуски нельзя было задним числом превращать в отдых.

## 11. Защита стрика

Рабочее название: `Streak Guard`.

### 11.1. Назначение

Защита предназначена для редкого непредвиденного пропуска. Она не должна заменять нормальное расписание отдыха.

### 11.2. Первая модель

- новый пользователь получает одну защиту;
- новая защита выдаётся за каждые 14 выполненных обязательных дней;
- одновременно можно хранить не более двух защит;
- защита применяется автоматически к первому подходящему пропуску;
- нельзя защитить более одного обязательного дня подряд;
- защищённый день не увеличивает стрик;
- защищённый день не даёт XP;
- Momentum уменьшается на `5`;
- расходуется одна защита.

### 11.3. Предварительно объявленный пропуск

Пользователь может заранее сообщить, что не сможет выполнить обязательство, но такой пропуск использует тот же ограниченный ресурс защиты.

Исключения вроде отпуска и болезни пока не проектируются.

## 12. Санкции

### 12.1. Запрещённые санкции

Система не должна:

- отнимать постоянный XP;
- понижать уровень;
- уничтожать достижения;
- создавать отрицательный XP;
- ежедневно увеличивать долг;
- заставлять отрабатывать весь перерыв;
- ухудшать прогресс других пользователей;
- блокировать базовые функции;
- использовать финансовые штрафы.

### 12.2. Защищённый пропуск

```text
Текущий стрик: сохраняется, но не увеличивается.
Momentum:      −5.
XP за день:     0.
Защита:        −1.
```

### 12.3. Первый незащищённый пропуск

```text
Текущий стрик: завершается.
Momentum:      −25.
Постоянный XP: без изменений.
Уровень:        без изменений.
```

### 12.4. Второй пропуск подряд

```text
Momentum: дополнительно −25.
```

### 12.5. Третий и последующие пропуски

Momentum может постепенно достигнуть `0`, после чего дополнительные числовые санкции прекращаются.

Система не должна бесконечно наказывать отсутствующего пользователя.

## 13. Восстановление

### 13.1. Быстрое восстановление однодневного разрыва

Доступно только:

- после одного незащищённого пропущенного дня;
- в течение следующих 48 часов;
- не чаще одного раза за 30 дней.

Предварительное условие:

```text
выполнить два минимальных полезных блока
```

Результат:

- предыдущий стрик восстанавливается;
- пропущенный день получает статус `repaired`;
- восстановление не даёт дополнительного XP;
- возвращается только часть потерянного Momentum;
- история показывает факт восстановления.

Пример:

```text
Стрик: 46 дней · 1 восстановление
```

### 13.2. Возвращение после длительного перерыва

После более долгого отсутствия:

- старый стрик остаётся в истории;
- начинается новый стрик;
- первый выполненный обязательный день даёт `+15 Momentum`;
- предлагается короткая восстановительная цель;
- накопленного долга нет;
- пропущенные дни не требуется отрабатывать.

## 14. Общая защита от grind

Система должна предотвращать:

- фиктивное минимальное действие ради стрика;
- бесконечное повторение дешёвого действия;
- резкое увеличение нагрузки ради множителя;
- использование защиты вместо расписания;
- заднее изменение режима;
- непропорциональное преимущество длинного стрика;
- отказ возвращаться после потери серии;
- постоянный страх отдыха;
- двойное начисление за одно событие;
- повторное получение lifetime-награды после reset или пересоздания.

Минимальные механизмы:

- доменные критерии полезности;
- содержательный порог дня;
- same-item и lifetime safeguards;
- domain-specific diminishing returns;
- при необходимости глобальный дневной envelope;
- ограниченный Momentum-бонус;
- ограниченный запас Streak Guard;
- отсутствие отрицательного XP;
- запланированный отдых;
- восстановление;
- прозрачное объяснение начислений.

## 15. Предварительно принятые решения

1. Уровень, стрик и Momentum являются разными сущностями.
2. Постоянный XP и уровень не уменьшаются.
3. Высокий уровень требует больше суммарного времени, а не бесконечного роста дневной нормы.
4. XP начисляется за подтверждённую работу, а не присутствие.
5. Каждый домен определяет собственные события полезного усилия.
6. Доменные проверки и антифарм-правила применяются до глобальных ограничений.
7. Стрик строится вокруг выбранных обязательных дней.
8. Запланированный отдых не является пропуском.
9. Стрик не даёт бесконечно растущий множитель.
10. Временный XP-бонус управляется ограниченным Momentum.
11. Пропуск снижает временные преимущества, но не уничтожает долгосрочный прогресс.
12. Защита стрика является ограниченным ресурсом.
13. После длительного перерыва нет накопленного долга.
14. Возвращение должно быть явно поддержано системой.
15. Anki является первым доменом детального проектирования.
16. Любая числовая формула должна проверяться симуляцией.

## 16. Параметры-гипотезы

Не считаются окончательными:

- 100 уровней первого цикла;
- формула `100 + 25 × L^1.2`;
- `200 Base XP` как продуктивный день;
- глобальный дневной diminishing returns;
- `50 Base XP` как минимум дня;
- изменения Momentum `+10`, `−5`, `−25`;
- максимальный Momentum-бонус `20%`;
- максимум двух Streak Guard;
- выдача защиты каждые 14 обязательных дней;
- восстановление в течение 48 часов;
- лимит одного восстановления за 30 дней;
- streak-вехи и их награды.

Доменные гипотезы Anki перечислены в [anki-xp-foundation.md](anki-xp-foundation.md).

## 17. Обязательная симуляция

Общая модель должна быть проверена минимум на сценариях:

1. Новый пользователь с низкой нагрузкой.
2. Стабильный пользователь с режимом `5/7`.
3. Пользователь, занимающийся каждый день.
4. Пользователь с длинными редкими сессиями.
5. Попытка фармить дешёвое действие.
6. Один случайный пропуск.
7. Два пропуска подряд.
8. Возвращение после недели отсутствия.
9. Возвращение после нескольких месяцев.
10. Многолетняя история и большой ретроспективный XP.
11. Регулярное использование максимума защиты.
12. Частое изменение расписания.
13. Несколько доменов с разными масштабами XP.
14. Двойное применение доменного и глобального diminishing returns.

Симуляция должна показать:

- время достижения ключевых уровней;
- распределение XP по доменам и источникам;
- вклад Momentum;
- эффект ограничений;
- частоту потери и восстановления стрика;
- наличие exploit-стратегий;
- риск того, что санкции делают возвращение невыгодным;
- чувствительность системы к разным режимам жизни.

## 18. Текущий следующий шаг

Первоначальный план предполагал сначала создать универсальную модель сравнения всех видов деятельности. После выбора Anki первым доменом порядок изменён.

Текущий следующий шаг:

> Подготовить Anki XP simulation specification и проверить Review, Learn и Create XP на синтетических и реальных профилях до фиксации абсолютных наград.

Нужно:

- выбрать нормализованную базовую единицу;
- сформировать профили разных типов;
- подготовить replay-наборы;
- сравнить варианты Review XP;
- проверить retrievability, memory gain и response-time factors;
- проверить volume bonus и same-day discount;
- смоделировать staged Learn XP;
- оценить Create XP и pending validation;
- сопоставить дневной XP с уровневой кривой;
- решить, нужен ли общий дневной diminishing returns поверх доменных ограничений.

После калибровки первого домена можно будет проектировать универсальные правила сопоставления Anki с другими областями жизни.
