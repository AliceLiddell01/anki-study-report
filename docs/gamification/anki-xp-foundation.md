# Anki XP Foundation

Статус: **DRAFT v0.2**  
Дата: **2026-07-16**  
Область: **продуктовый концепт первого домена системы игрофикации; реализация в Anki Study Report пока не проектируется**

## 1. Назначение документа

Этот документ фиксирует общий фундамент начисления опыта за полезную работу в Anki.

Anki является первым доменом будущей системы прогрессии, поскольку предоставляет подробную историю учебной деятельности и позволяет проектировать награду на основе проверяемых событий.

Основной принцип:

> Опыт начисляется за подтверждённую учебную работу и полезное изменение состояния памяти или материала, а не за время, проведённое в приложении.

Система не должна вознаграждать:

- открытое окно Anki;
- простой таймер активности;
- нахождение карточки на экране;
- бессодержательные нажатия;
- повторяемые дешёвые операции;
- действия, созданные исключительно ради игрового XP.

## 3. Три режима Anki XP

```text
Anki XP
├── Review XP   — повторение уже изученного материала;
├── Learn XP    — первоначальное освоение нового материала;
└── Create XP   — создание, исправление и подготовка карточек.
```

Эти режимы используют разные единицы полезной работы и не должны рассчитываться одной универсальной формулой.

### 3.1. Review XP

Вознаграждает попытку воспроизвести уже изученное знание и полезное усиление памяти.

### 3.2. Learn XP

Вознаграждает не сам показ новой карточки, а прохождение этапов первоначального освоения и последующее подтверждение памяти.

### 3.3. Create XP

Вознаграждает полезные переходы состояния учебного материала: создание пригодной карточки, устранение проблемы и подтверждённое улучшение.

## 4. Термины

### 4.1. Raw reward

Предварительная ценность события до антифарм-проверок, дисконтирования повторов и дневных ограничений.

### 4.2. Valid event

Событие, которое прошло проверки происхождения, состояния карточки, времени, повторяемости и допустимого режима.

### 4.3. Immediate XP

Опыт, который может быть подтверждён сразу после события.

### 4.4. Pending XP

Опыт, который резервируется и подтверждается только последующим результатом.

Примеры:

- новая карточка успешно воспроизведена на следующий день;
- созданная заметка действительно вошла в обучение;
- исправленная проблемная карточка стала пригоднее.

### 4.5. Lifetime reward state

История уже полученных наград, связанная с карточкой, заметкой, полем или переходом состояния.

Она предотвращает повторное получение одной награды через:

- reset;
- удаление и повторное создание;
- очистку и повторное заполнение поля;
- многократное сохранение;
- импорт тех же данных;
- повторную обработку истории.

### 4.6. Productive Anki work

Работа, которая относится хотя бы к одному из трёх режимов и прошла доменные проверки полезности.

### 4.7. Review Unit

Промежуточная относительная единица ценности Review Episode.

```text
1.00 Review Unit
≈
обычное успешное плановое повторение
```

`Review Unit` конвертируется в глобальный XP только после симуляции и общей калибровки экономики.

## 5. Общие принципы Anki XP

### 5.1. Результат важнее присутствия

Система оценивает:

- попытку извлечения знания из памяти;
- прогнозируемую сложность воспроизведения;
- изменение состояния памяти;
- прохождение этапов первоначального обучения;
- переход материала к учебной готовности;
- устранение подтверждённой проблемы.

Время используется только как дополнительный сигнал достоверности. Само по себе время не создаёт XP.

### 5.2. Одинаковое действие не обязано иметь одинаковую ценность

Две карточки могут давать разную награду, если различаются:

- вероятностью воспроизведения перед ответом;
- стадией обучения;
- прогнозируемым усилением памяти;
- происхождением события;
- достоверностью ответа;
- количеством предыдущих предъявлений;
- полезностью созданного или исправленного материала.

### 5.3. Награда не должна управлять кнопкой ответа

Пользователь должен выбирать `Again`, `Hard`, `Good` или `Easy` по фактическому качеству воспроизведения, а не по размеру XP.

Следовательно:

- `Hard`, `Good` и `Easy` не имеют прямого различия игровой награды;
- memory gain стандартизируется через counterfactual `Good`;
- `Again` получает ограниченную награду за честную попытку;
- злоупотребление оценками рассматривается отдельной abuse-моделью.

### 5.4. Ошибка не равна отсутствию работы

Неудачное воспроизведение даёт небольшую награду за честную проверку памяти.

Однако:

- challenge bonus не начисляется;
- memory gain не начисляется;
- relearning получает только ограниченный support credit;
- повторные ошибки не могут бесконечно увеличивать XP.

### 5.5. Полезная работа имеет предел игровой отдачи

Длинная сессия остаётся полезной, но:

- повторные показы одной карточки дисконтируются;
- массовые ранние reviews не равны плановым;
- relearning имеет episode cap;
- объёмный бонус ограничен;
- однообразный grind получает уменьшающуюся отдачу;
- доменные ограничения применяются до общего дневного diminishing returns.

### 5.6. Отсутствие FSRS не лишает базовой награды

FSRS позволяет точнее оценить контекст памяти, но не определяет само право пользователя получать XP.

При отсутствии надёжной модели:

- сохраняются попытка и исход;
- используется нейтральный контекст;
- точные challenge и memory-gain bonuses отключаются или смешиваются с нейтральным значением.

## 6. Целевая структура экономики

На длинном промежутке ожидаемая структура Anki XP предварительно выглядит так:

| Канал | Ориентировочный вклад |
|---|---:|
| `Review XP` | `55–70%` |
| `Learn XP` | `25–40%` |
| `Create XP` | `5–10%` |

Это не дневные квоты.

Пользователь может:

- проводить день без новых карточек;
- изучать только новый материал;
- заниматься исправлением коллекции;
- комбинировать все режимы.

Диапазоны нужны для проверки того, не стала ли одна механика доминировать над всей прогрессией.

## 7. Review XP Foundation

### 7.1. Награждаемая единица

Единицей является `Review Episode`, а не строка `revlog`.

```text
Review Episode
=
первичная попытка воспроизвести уже изученную карточку
+
связанные с ней шаги восстановления после ошибки
```

Не более одного основного core-эпизода допускается на одну card ID за один учебный день Anki.

Полная классификация событий описана в [anki-review-event-taxonomy.md](anki-review-event-taxonomy.md).

### 7.2. Классы событий

```text
core
support
supplemental
none
route_to_learn
deferred
```

Примеры:

| Событие | Класс |
|---|---|
| Первое плановое due-review | `core` |
| Первое overdue-review | `core` |
| Первичный `Again` | `core`, fail outcome |
| Relearning после `Again` | `support` |
| Early review | `supplemental` |
| Preview без rescheduling | `none` или отдельный Practice Credit |
| Manual/reset/reschedule | `none` |
| New/learning | `route_to_learn` |

### 7.3. Актуальная модель награды

Ранняя мультипликативная формула больше не используется.

Актуальная модель описана в [anki-review-reward-model.md](anki-review-reward-model.md):

```text
CoreReviewUnits =
    ResponseValidity
  × (
        AttemptCredit
      + Pass × OutcomeCredit
      + Pass × ContextCredit
    )
```

Контекст включает:

- retrieval challenge;
- ограниченный memory gain;
- model confidence;
- backlog protection.

### 7.4. Кандидатный масштаб

```text
обычный успешный core Episode: около 1.00 Review Unit
честный Again:                 около 0.25 Review Unit
трудный успешный Episode:      около 1.10–1.30 Review Unit
```

Точные значения являются гипотезами.

### 7.5. FSRS-сигналы

Используются:

- `Retrievability`;
- `Stability`;
- `Difficulty`;
- counterfactual transition при `Good`;
- уверенность модели.

`Difficulty` не даёт отдельный XP-бонус. Она уже участвует в моделировании памяти и остаётся аналитическим сигналом.

### 7.6. Retrievability

Успешное воспроизведение при более низкой `Retrievability` может давать больший challenge credit.

Но:

- бонус имеет максимум;
- при экстремально низкой вероятности он снижается;
- дополнительная выгода просрочки дисконтируется;
- forced-due и manual-reschedule ограничиваются таксономией;
- `Again` не получает challenge bonus.

### 7.7. Memory gain

Memory gain рассчитывается по стандартизированному успешному сценарию:

```text
Hard / Good / Easy
→ counterfactual Good transition
```

Это не позволяет получать больше XP простым выбором `Easy`.

Возможные модели нормализации:

- фиксированная логарифмическая шкала;
- percentile внутри сопоставимых cohorts;
- полный отказ от отдельного gain-компонента.

Выбор должен сделать симулятор.

### 7.8. Время ответа

Время не создаёт положительный бонус.

Оно может:

- подтвердить правдоподобность события;
- уменьшить награду за физически невозможный ответ;
- заметить серию экстремально быстрых нажатий;
- ограничить событие, достигшее timer cap;
- учитывать особенности audio, cloze и production templates.

### 7.9. Relearning

Relearning не повторяет core-формулу.

Используется ограниченный support credit на весь lapse episode.

Первая гипотеза:

```text
support cap ≈ 0.12 Review Unit
полный Again + relearning episode ≈ не более 0.37
```

### 7.10. Explainability

Каждый расчёт должен сохранять:

```text
event class
attempt credit
outcome credit
challenge credit
memory gain credit
confidence adjustment
response validity
final Review Units
rule versions
```

## 8. Learn XP Foundation

### 8.1. Основной принцип

Полная награда не выдаётся при первом показе новой карточки.

```text
первое знакомство
≠
подтверждённое изучение
```

Награда распределяется по этапам и частично хранится как `Pending XP`.

### 8.2. Кандидатные этапы

| Этап | Предварительная доля acquisition reward |
|---|---:|
| Первое осмысленное знакомство | `15–20%` |
| Завершение initial learning | `25–30%` |
| Первое успешное междневное воспроизведение | `30–35%` |
| Раннее подтверждение закрепления | `20–25%` |

Конкретная карточка имеет ограниченную lifetime acquisition reward.

### 8.3. Почему используется Pending XP

Без отложенной проверки пользователь мог бы:

1. добавить большое количество новых карточек;
2. один раз их показать;
3. получить полную награду;
4. не закрепить материал.

Pending-модель связывает значительную часть XP с последующим подтверждением памяти.

### 8.4. Learning efficiency

Эффективность может учитывать:

- количество попыток;
- число `Again`;
- длительность относительно ожидаемой;
- выпуск из learning;
- первое междневное воспроизведение;
- раннее закрепление;
- рост stability.

Неэффективное обучение получает меньше, но не ноль. Сложный материал не должен считаться бесполезным.

### 8.5. Массовое введение новых карточек

Immediate XP получает diminishing returns при резком превышении нормального объёма.

Но pending reward за последующее реальное закрепление может подтверждаться полностью.

Это ограничивает фарм вводом, не штрафуя качественно выученный материал.

## 9. Create XP Foundation

### 9.1. Основной принцип

Creation XP начисляется за переходы состояния материала, а не за операции редактора.

```text
Draft
→ Structurally Complete
→ Study Ready
→ Used in Learning
→ Validated
```

### 9.2. Draft

Заметка существует, но не готова к обучению.

```text
XP: 0
```

### 9.3. Structurally Complete

Заполнены обязательные поля и отсутствуют очевидные структурные ошибки.

```text
XP: очень низкий
```

### 9.4. Study Ready

Материал соответствует правилам note type:

- обязательные поля заполнены;
- вопрос и ответ различаются;
- cloze валиден;
- media существует;
- HTML не сломан;
- отсутствует точный дубликат.

```text
XP: низкий предварительный
```

### 9.5. Used in Learning

Карточка реально вошла в учебный процесс.

```text
XP: дополнительная подтверждённая часть
```

### 9.6. Validated

Материал оказался полезен после использования:

- карточка не была сразу удалена;
- изменение не откатили;
- leech был исправлен;
- проблема перестала повторяться;
- карточка успешно изучается.

```text
XP: отложенная подтверждённая часть
```

### 9.7. Meaningful Edit

Не засчитываются:

- пробелы;
- повторное сохранение;
- форматирование без изменения содержания;
- изменение и немедленный возврат;
- очистка и повторное заполнение одного поля;
- массовые служебные операции;
- повторное создание уже награждённого материала.

Могут засчитываться:

- первое валидное заполнение обязательного поля;
- добавление отсутствующего ответа;
- исправление cloze;
- устранение broken media;
- существенная переформулировка проблемной карточки;
- подтверждённое исправление leech.

### 9.8. Ограничение роли Create XP

Предварительно:

```text
Create XP ≈ 5–10% долгосрочной экономики
```

В обычный день создание не должно полностью заменять работу памяти.

В будущем возможен отдельный запланированный creation day, но он всё равно требует доменных caps и подтверждения результата.

## 10. Общий pipeline Anki XP

```text
Anki event
→ source classification
→ Review / Learn / Create routing
→ eligibility and provenance checks
→ channel-specific reward model
→ per-card / per-note lifetime safeguards
→ same-item repeat limits
→ channel diminishing returns
→ aggregation into Anki Base XP
→ global daily envelope
→ Momentum bonus
→ final global XP
```

Порядок важен.

Нельзя сначала начислить полный XP, а затем пытаться исправить farm только глобальным дневным cap.

## 11. Универсальные anti-farm правила

1. Одна строка `revlog` не равна одной полной награде.
2. Одна карточка не создаёт несколько core Review Episodes за день.
3. Relearning имеет общий cap.
4. Early review не равен плановому review.
5. Preview без rescheduling не получает полный постоянный XP.
6. Manual, reset и reschedule записи не являются учебной работой.
7. Нажатие `Easy` не повышает игровой memory gain.
8. Долгое ожидание не создаёт XP.
9. Экстремально быстрые ответы проходят validity check.
10. Массовое введение нового получает diminishing returns.
11. Полная Learn-награда требует последующего подтверждения.
12. Creation XP связан с переходом состояния и lifetime reward state.
13. Удаление и повторное создание не обнуляет историю наград.
14. Undo отменяет provisional reward.
15. Повторная обработка одного события должна быть идемпотентной.
16. Каждый расчёт хранит версию правил и breakdown.

## 12. Доступные и требующие проверки данные

Anki history предоставляет базовые сигналы:

- card ID;
- выбранную оценку;
- интервалы;
- время ответа;
- тип review-события;
- timestamp.

Для полной игровой модели дополнительно могут потребоваться:

- состояние карточки до ответа;
- `Retrievability` до ответа;
- `Stability` и `Difficulty`;
- natural due reference;
- происхождение rescheduling;
- sibling exposure;
- note/card template context;
- признак Undo;
- model confidence;
- lifetime reward state.

Не следует утверждать, что всё это можно надёжно восстановить из одной строки `revlog`. Техническая доступность должна быть проверена отдельно до реализации.

## 13. Симуляция перед реализацией

Нужно проверить минимум:

### Review

- своевременные reviews;
- умеренный и тяжёлый backlog;
- `Again`, `Hard`, `Good`, `Easy`;
- repeated relearning;
- early и filtered reviews;
- forced due;
- siblings;
- разные desired retention;
- отсутствие FSRS;
- малая история пользователя;
- разные версии FSRS;
- быстрый spam и долгие паузы.

### Learn

- маленький стабильный ввод;
- массовый ввод;
- успешное раннее закрепление;
- частые ошибки;
- reset;
- interday learning;
- abandonment после первого показа.

### Create

- полноценное создание;
- пустые и неполные заметки;
- повторное сохранение;
- очистка и заполнение полей;
- импорт;
- массовые теги;
- исправление leech;
- удаление и повторное создание;
- broken media;
- sibling generation.

## 14. Предварительно принятые решения

1. Anki XP разделён на Review, Learn и Create.
2. Время само по себе не является награждаемой работой.
3. Review использует `Review Episode`, а не строку `revlog`.
4. Review Reward Model является аддитивной ограниченной моделью.
5. `Again` получает небольшую награду за попытку.
6. Успешные кнопки не имеют прямой разницы XP.
7. Retrievability может повышать reward только с caps и backlog protection.
8. Memory gain стандартизируется через counterfactual `Good`.
9. Difficulty не получает отдельный положительный коэффициент.
10. Response time используется только для validity.
11. Relearning получает ограниченный support credit.
12. Без FSRS сохраняется нейтральная базовая награда.
13. Learn XP распределяется во времени.
14. Creation XP начисляется за полезные переходы состояния.
15. Все каналы используют lifetime и duplicate safeguards.
16. Числовые параметры становятся правилами только после симуляции.

- ретроспективный расчёт старой истории.
