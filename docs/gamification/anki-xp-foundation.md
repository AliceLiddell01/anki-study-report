# Anki XP Foundation

Статус: **DRAFT v0.1**  
Дата: **2026-07-16**  
Область: **продуктовый концепт первого домена системы игрофикации; реализация в Anki Study Report пока не проектируется**

## 1. Назначение документа

Этот документ фиксирует фундамент начисления опыта за полезную работу в Anki.

Anki является первым доменом будущей системы прогрессии. Он используется не потому, что вся система должна навсегда остаться привязанной к Anki, а потому, что Anki предоставляет достаточно подробную историю реальной учебной деятельности и позволяет начать проектирование с проверяемых событий.

Основной принцип:

> Опыт начисляется за подтверждённую учебную работу и полезное изменение состояния материала, а не за время, проведённое в приложении.

Система не должна вознаграждать:

- открытое окно Anki;
- простой таймер активности;
- нахождение карточки на экране;
- бессодержательные нажатия;
- повторяемые дешёвые операции;
- действия, созданные исключительно ради игрового XP.

Документ разделяет Anki XP на три режима:

```text
Anki XP
├── Review XP   — повторение уже изученного материала;
├── Learn XP    — первоначальное освоение нового материала;
└── Create XP   — создание, исправление и подготовка карточек.
```

Числовые значения ниже являются параметрами первой модели. Они не считаются окончательными до симуляции на реальных и синтетических историях обучения.

## 2. Связь с общим фундаментом

Общая прогрессия, уровни, стрик, `Momentum`, пропуски и санкции описаны в [progression-foundation.md](progression-foundation.md).

Этот документ отвечает только за переход:

```text
события Anki → валидированное полезное усилие → Anki Base XP
```

Он не определяет окончательно:

- кривую уровней;
- отображение XP;
- награды за уровни;
- навыки;
- достижения;
- квесты;
- игровую валюту;
- UI;
- техническое хранилище;
- API;
- интеграцию в production-код.

## 3. Термины

### 3.1. Raw reward

Предварительная ценность события до антифарм-проверок, дисконтирования повторов и дневных ограничений.

### 3.2. Valid event

Событие, которое прошло проверки происхождения, состояния карточки, времени, повторяемости и допустимого режима.

### 3.3. Immediate XP

Опыт, который можно выдать сразу после события.

### 3.4. Pending XP

Опыт, который резервируется, но подтверждается только последующим результатом.

Примеры:

- новая карточка успешно воспроизведена на следующий день;
- созданная заметка действительно вошла в обучение;
- исправленная проблемная карточка стала пригоднее.

### 3.5. Lifetime reward state

История уже полученных наград, связанная с карточкой, заметкой, полем или переходом состояния. Она нужна, чтобы удаление, сброс, повторное заполнение и повторное создание не позволяли многократно получать одну и ту же награду.

### 3.6. Productive Anki work

Работа, которая относится хотя бы к одному из трёх режимов и прошла доменные проверки полезности.

## 4. Общие принципы Anki XP

### 4.1. Результат важнее присутствия

Система оценивает:

- попытку извлечения знания из памяти;
- изменение прогнозируемого состояния памяти;
- прохождение этапов первоначального обучения;
- переход материала к учебной готовности;
- устранение подтверждённой проблемы.

Время используется только как дополнительный сигнал достоверности и усилия. Само по себе время не создаёт XP.

### 4.2. Одинаковое действие не обязано иметь одинаковую ценность

Две карточки могут давать разный XP, если различаются:

- вероятностью воспроизведения перед ответом;
- сложностью;
- ожидаемым приростом устойчивости памяти;
- стадией обучения;
- качеством и достоверностью события;
- количеством предыдущих предъявлений в тот же день.

### 4.3. Награда не должна управлять ответом пользователя

Игровая система не должна делать конкретную кнопку Anki выгоднее другой.

Пользователь должен выбирать `Again`, `Hard`, `Good` или `Easy` по фактическому качеству воспроизведения, а не по размеру XP.

### 4.4. Ошибка не равна отсутствию работы

Неудачное воспроизведение может дать небольшую награду за честную попытку и последующее переобучение. Однако полная награда предназначена для успешного извлечения знания и усиления памяти.

### 4.5. Полезная работа имеет предел игровой отдачи

Длинная сессия остаётся полезной, но:

- повторные показы одной карточки дисконтируются;
- массовые ранние reviews не равны плановым;
- объёмный бонус ограничен;
- однообразный grind имеет уменьшающуюся отдачу;
- доменные ограничения применяются до общего дневного diminishing returns.

## 5. Целевая структура экономики

На длинном промежутке ожидаемая структура Anki XP предварительно выглядит так:

| Канал | Ориентировочный вклад |
|---|---:|
| `Review XP` | `55–70%` |
| `Learn XP` | `25–40%` |
| `Create XP` | `5–10%` |

Это не дневные обязательные квоты.

Пользователь может проводить день без новых карточек или отдельный день подготовки материала. Диапазоны нужны для проверки того, не стала ли одна механика доминировать над всей прогрессией.

## 6. Review XP

### 6.1. Что считается Review

`Review XP` относится к проверке уже изученного материала, включая обычное повторение и ограниченную часть relearning после ошибки.

Основным событием считается первое валидное плановое предъявление карточки за учебный день.

Повторные внутридневные предъявления могут учитываться, но получают значительно меньший вес.

### 6.2. Доступные сигналы FSRS

Для оценки используются три основные характеристики FSRS:

- `Retrievability` — прогнозируемая вероятность успешного воспроизведения в текущий момент;
- `Stability` — устойчивость памяти, связанная со скоростью снижения retrievability;
- `Difficulty` — относительная сложность материала для конкретного пользователя.

Следует использовать формулировку «низкая прогнозируемая вероятность воспроизведения», а не утверждение, что карточка «точно забудется». FSRS оценивает вероятность, а не гарантированный исход.

### 6.3. Концептуальная формула

Первая модель:

```text
ReviewRawXP =
    BaseReviewReward
  × OutcomeFactor
  × RetrievalChallengeFactor
  × MemoryGainFactor
  × DifficultyFactor
  × ResponseValidityFactor
  × ModeValidityFactor
```

После расчёта применяются:

```text
same-day repeat discount
→ per-card safeguards
→ volume bonus
→ channel diminishing returns
→ global daily envelope
```

Ни один коэффициент не считается окончательно откалиброванным.

### 6.4. Base Review Reward

`BaseReviewReward` — единица ценности первого валидного планового повторения.

Её абсолютное значение будет определено только после симуляции общей кривой уровней. На этапе концепта важнее относительные коэффициенты.

### 6.5. Outcome Factor

Предварительная модель:

| Исход | Коэффициент |
|---|---:|
| Не удалось воспроизвести; выбран `Again` | `0.30–0.40` |
| Успешное воспроизведение | `1.00` |

`Again` не даёт ноль, потому что честная попытка и relearning являются частью учебной работы.

`Hard`, `Good` и `Easy` не должны напрямую давать разные базовые награды. Иначе выбор кнопки станет игровой стратегией.

Дополнительная аналитика кнопки допускается только как ограниченный сигнал достоверности и не должна делать `Easy` выгоднее.

### 6.6. Retrieval Challenge Factor

Успешное воспроизведение при более низкой `Retrievability` должно давать больше XP, поскольку задача объективно была сложнее.

Первая шкала:

| Retrievability перед ответом | Коэффициент сложности |
|---:|---:|
| `0.90–1.00` | `1.00` |
| `0.75–0.90` | `1.10` |
| `0.50–0.75` | `1.25` |
| `0.20–0.50` | `1.40` |
| `< 0.20` | `1.35` |

Коэффициент намеренно перестаёт расти на экстремально низкой retrievability.

Причина: линейный рост награды сделал бы намеренное накопление backlog и откладывание карточек выгодной стратегией.

### 6.7. Защита от фарма просрочкой

Низкая retrievability сама по себе не гарантирует полный challenge bonus.

Дополнительный бонус должен учитывать характер просрочки:

| Состояние | Доступная доля challenge bonus |
|---|---:|
| Повторение около планового срока | до `100%` |
| Умеренная просрочка | около `75%` |
| Сильная просрочка | около `40–50%` |

Базовый XP не отнимается. Ограничивается только выгода от намеренного доведения карточки до почти полного забывания.

### 6.8. Memory Gain Factor

Полезность повторения выражается не только сложностью извлечения, но и ожидаемым усилением памяти.

Предлагаемый сигнал:

```text
нормализованный ожидаемый прирост Stability
```

Чтобы пользователь не мог увеличивать XP нажатием `Easy`, игровая модель должна использовать стандартизированный сценарий успешного ответа.

Рабочая гипотеза:

> Для расчёта игрового прироста используется контрфактическое значение, соответствующее честному успешному `Good`, независимо от фактически выбранной успешной кнопки.

Предварительный диапазон:

```text
MemoryGainFactor = 0.90–1.25
```

Retrievability и memory gain частично коррелируют, поэтому оба коэффициента не должны одновременно получать большой вес.

### 6.9. Difficulty Factor

`Difficulty` допускается как небольшой корректирующий коэффициент:

```text
DifficultyFactor = 0.95–1.15
```

Сильный бонус за difficulty запрещён, потому что он может сделать перегруженные, двусмысленные и плохо составленные карточки выгоднее качественных.

### 6.10. Время ответа

Время ответа не является самостоятельной наградой.

Запрещённые модели:

```text
дольше отвечал → больше XP;
быстрее отвечал → больше XP.
```

Обе модели легко эксплуатируются и плохо отражают полезность.

Вместо абсолютного времени используется персональная ожидаемая длительность:

```text
ResponseRatio = ActualResponseTime / ExpectedResponseTime
```

`ExpectedResponseTime` может учитывать:

- медиану предыдущих успешных ответов;
- тип заметки и шаблон карточки;
- наличие аудио;
- формат вопроса;
- историю конкретной карточки;
- `Review` или `Relearn`;
- retrievability перед ответом.

Предварительная интерпретация:

| Поведение | Response Validity Factor |
|---|---:|
| Подозрительно быстрый ответ, `< 20%` ожидаемого | `0.25–0.50` |
| Быстрый, но правдоподобный ответ | `1.00–1.05` |
| Нормальный диапазон | `1.00` |
| Заметно медленнее ожидаемого | `0.90–1.00` |
| Экстремально долгий ответ | `0.60–0.80` |

Максимальный бонус за скорость должен оставаться маленьким, ориентировочно не более `5%`.

Более низкая retrievability расширяет допустимый диапазон времени. Трудное воспроизведение закономерно может занимать дольше.

### 6.11. Ограничение записанного времени

Экстремально большое время не должно увеличивать XP. Оно может означать:

- паузу;
- отвлечение;
- уход от компьютера;
- разбор внешнего материала;
- искусственное ожидание.

Нужно учитывать фактический лимит времени ответа, записываемого Anki, и не пытаться извлекать ложную точность из значения, достигшего cap.

### 6.12. Объёмный бонус

Количество повторений должно давать ограниченный boost, но только после фильтрации валидных событий.

Считаются прежде всего уникальные карточки с первым плановым review за учебный день.

Первая шкала:

| Уникальные валидные reviews за день | Бонус |
|---:|---:|
| `1–24` | `0%` |
| `25–49` | `+3%` |
| `50–99` | `+6%` |
| `100–199` | `+10%` |
| `200+` | `+12–15%` |

Бонус:

- имеет жёсткий cap;
- не перемножается сам с собой;
- не считает одну карточку несколько раз полностью;
- не превращает ранние reviews в эквивалент due reviews;
- рассчитывается после проверки достоверности событий.

### 6.13. Повторные показы одной карточки

Предварительная модель:

| Событие | Доля обычного XP |
|---|---:|
| Первое валидное плановое предъявление за день | `100%` |
| Первое relearning-предъявление после `Again` | `20–30%` |
| Последующие внутридневные показы | `5–15%` |

Точные коэффициенты зависят от learning steps и требуют симуляции.

### 6.14. Filtered decks, cram и ранние reviews

События должны классифицироваться по режиму.

Предварительно:

- плановый due review получает полный доступный XP;
- review немного раньше срока получает небольшой discount;
- значительно ранний review получает сильный discount;
- filtered/cram не считается автоматически равным обычному review;
- повторение без изменения scheduling требует отдельной политики;
- preview и просмотр ответа без валидного revlog-события не дают XP.

Система не должна делать многократный ручной прогон знакомых карточек оптимальным способом фарма.

### 6.15. Completion Bonus

Допускается небольшой бонус за завершение доступной плановой очереди.

Он должен учитывать:

- реальные дневные лимиты;
- выбранный scope;
- backlog;
- наличие новых карточек;
- пользовательский режим.

Нельзя требовать очищать огромный backlog целиком, если это вредно для режима обучения.

## 7. Learn XP

### 7.1. Что считается изучением нового

`Learn XP` относится к первоначальному освоению карточки, которая ранее не имела подтверждённой истории изучения.

Первый показ не означает, что материал выучен. Поэтому полная награда не выдаётся сразу.

### 7.2. Acquisition reward

Каждая новая карточка получает ограниченный максимальный `AcquisitionReward`, распределённый по этапам.

Первая модель:

| Этап | Доля награды |
|---|---:|
| Первое осмысленное знакомство | `15–20%` |
| Завершение внутридневного обучения | `25–30%` |
| Первое успешное междневное воспроизведение | `30–35%` |
| Подтверждение раннего закрепления | `20–25%` |

Пример для условной карточки стоимостью `30 XP`:

```text
6 XP  — первое изучение;
8 XP  — завершение learning steps;
10 XP — успешное первое междневное воспроизведение;
6 XP  — ранний checkpoint закрепления.
```

Абсолютное значение `30 XP` является только примером.

### 7.3. Pending XP

Большая часть Learn XP хранится как `Pending XP` до подтверждения результата.

Это защищает от стратегии:

```text
массово добавить новые карточки
→ один раз показать их
→ получить полную награду
→ не закреплять материал
```

Невыплаченная часть не является штрафом. Она просто ещё не заработана.

### 7.4. Learning Efficiency

Эффективность первоначального освоения может учитывать:

- количество попыток;
- количество `Again`;
- время относительно ожидаемого;
- завершение learning steps;
- первое междневное воспроизведение;
- ранний рост Stability;
- сохранение карточки в обучении.

Предварительный диапазон:

```text
LearningEfficiencyFactor = 0.75–1.15
```

Неэффективное обучение не должно давать ноль. Сложный материал закономерно может потребовать больше попыток.

### 7.5. Не поощрять кнопку Easy

Фактический выбор `Easy` не должен сразу разблокировать всю награду за новую карточку.

Этапы Learn XP подтверждаются последующими событиями и состоянием памяти, а не только кнопкой первого ответа.

### 7.6. Массовое введение новых карточек

Новые карточки создают будущую review-нагрузку. Поэтому immediate-часть Learn XP получает diminishing returns.

Первая концепция:

```text
первые N новых карточек в нормальном диапазоне: 100% immediate XP;
следующий диапазон:                        70%;
значительное превышение устойчивой нормы: 30–40%.
```

`N` не должно быть одинаковым для всех. Оно может зависеть от:

- среднего количества новых карточек;
- текущей review-нагрузки;
- backlog;
- retention;
- выбранного режима;
- истории успешного закрепления.

Отложенный XP за последующее реальное усвоение может выдаваться полностью даже при массовом вводе. Ограничивается прежде всего выгода от самого факта добавления большого объёма нового материала.

### 7.7. Повторное создание и сброс

Карточка не должна повторно получать полный Acquisition Reward после:

- ручного reset;
- forget;
- перемещения;
- смены колоды;
- удаления и восстановления;
- повторного импорта;
- клонирования идентичной заметки.

Для этого нужен lifetime reward state и политика идентификации повторно созданного материала.

## 8. Create XP

### 8.1. Назначение

`Create XP` вознаграждает подготовку и улучшение учебного материала.

Его вклад намеренно значительно ниже Review XP и Learn XP, потому что автоматическая система может достаточно надёжно проверить структуру, но не всегда способна доказать смысловое качество карточки.

### 8.2. Не считать операции редактора

XP не начисляется напрямую за:

- нажатия клавиш;
- время в редакторе;
- количество сохранений;
- добавление пробелов;
- перестановку HTML без смыслового изменения;
- массовую смену тегов;
- перемещение карточек;
- очистку и повторное заполнение поля;
- создание случайного текста.

### 8.3. Состояния материала

Предлагаемая модель:

```text
Draft
  ↓
Structurally Complete
  ↓
Study Ready
  ↓
Used in Learning
  ↓
Validated
```

#### Draft

Материал существует, но не готов к обучению.

```text
XP: 0
```

#### Structurally Complete

Заполнены обязательные поля, создаются ожидаемые карточки, отсутствуют очевидные структурные ошибки.

```text
XP: очень низкий
```

#### Study Ready

Заметка соответствует проверяемым правилам note type:

- обязательные поля непустые;
- вопрос и ответ не являются идентичными;
- cloze имеет валидное удаление;
- media существует;
- нет очевидно сломанного HTML;
- материал не является точным дубликатом.

```text
XP: небольшая предварительная награда
```

#### Used in Learning

Карточка реально была введена в обучение.

```text
XP: дополнительная небольшая часть
```

#### Validated

Материал подтвердил пригодность:

- карточка не была немедленно удалена;
- изменение не было сразу отменено;
- карточка используется;
- исправленная проблема перестала воспроизводиться;
- leech после переработки стал успешно вспоминаться.

```text
XP: отложенная подтверждённая часть
```

### 8.4. Meaningful Edit

Содержательным может считаться переход, который изменяет нормализованное состояние материала и проходит проверку типа изменения.

Потенциально допустимы:

- первое заполнение обязательного пустого поля;
- добавление отсутствующего ответа;
- исправление broken media;
- исправление cloze-структуры;
- существенная переформулировка вопроса или ответа;
- разделение перегруженной карточки;
- устранение подтверждённой проблемы;
- переработка leech-карточки.

Не засчитываются:

- whitespace-only изменения;
- повторное сохранение;
- изменение и немедленный возврат;
- эквивалентная перестановка HTML;
- повторное очищение и заполнение уже награждённого поля;
- массовые служебные изменения без учебного результата.

### 8.5. Пример относительных наград

| Событие | Относительная награда |
|---|---:|
| Простое сохранение заметки | `0` |
| Первое заполнение обязательного поля | очень низкая |
| Первое достижение структурной готовности | низкая |
| Создание полноценной новой заметки | низкая |
| Добавление корректного media | очень низкая |
| Содержательное исправление вопроса или ответа | низкая |
| Исправление broken media или HTML | низкая |
| Переработка leech-карточки | умеренная |
| Подтверждённое улучшение проблемной карточки | умеренная |
| Массовая смена тегов или колоды | `0` или почти `0` |
| Импорт готовой колоды | `0` либо отдельная будущая политика |

### 8.6. Относительное соотношение с Review

Первая гипотеза:

```text
обычное успешное Review:          1.00 единица;
полный Acquisition новой карточки: 2.50–4.00 единицы во времени;
простое полезное редактирование:  0.05–0.20 единицы;
подтверждённое исправление:       0.30–0.70 единицы.
```

Эти отношения требуют проверки на пользователях, которые создают карточки вручную, импортируют готовые колоды и активно обслуживают большие коллекции.

### 8.7. Immediate и Pending Create XP

Первая модель:

```text
30% — после автоматически проверяемого полезного перехода;
70% — после использования или подтверждения результата.
```

Разделение не обязано быть одинаковым для каждого типа редактирования.

Например, исправление отсутствующего media может подтверждаться сразу, а смысловая переработка leech требует последующего учебного результата.

### 8.8. Lifetime reward state полей

Каждое награждаемое поле и переход состояния должны помнить, что награда уже была выдана.

Пример:

```text
поле уже получило XP за первое валидное заполнение
→ очистка и повторное заполнение не дают новый XP
```

### 8.9. Дневной вклад Create XP

Предварительное ограничение:

```text
Create XP ≤ 10–15% обычного дневного XP
```

Это ограничение пока является гипотезой.

Допускается будущий отдельный режим «день создания материала», в котором создание может засчитать активный день. Даже в таком режиме оно не должно автоматически приносить столько же XP, сколько полноценная работа памяти.

## 9. Взаимодействие трёх каналов

### 9.1. Порядок расчёта

Рекомендуемый концептуальный pipeline:

```text
Anki event
→ classification: Review / Learn / Create
→ source validation
→ channel-specific reward
→ same-item and lifetime safeguards
→ channel-specific diminishing returns
→ aggregation into Anki Base XP
→ global daily diminishing returns
→ Momentum bonus
→ final XP ledger entry
```

### 9.2. Не суммировать несколько полных наград за одно событие

Одна операция может иметь несколько смыслов, но не должна создавать несколько полных наград.

Пример:

```text
первая междневная попытка новой карточки
```

Она может одновременно:

- завершать этап Learn XP;
- быть полноценным Review-событием.

Нужно определить либо приоритет, либо комбинированную награду с cap, чтобы одно действие не оплачивалось дважды полностью.

### 9.3. Стрик и статус дня

Anki-день может считаться `complete`, если достигнут содержательный порог подтверждённой работы.

Порог не должен состоять только из количества событий.

Предварительно в него могут входить:

- валидный Review XP;
- подтверждённые этапы Learn XP;
- ограниченная доля Create XP.

Создание не должно позволять сохранять стрик бесконечным редактированием одного поля.

Точная формула зачёта Anki-дня является отдельной задачей симуляции.

## 10. Универсальные антифарм-правила

1. Первое валидное событие по карточке за день имеет наибольший вес.
2. Повторные внутридневные показы резко дисконтируются.
3. `Easy` не увеличивает игровой XP напрямую.
4. Долгое ожидание на карточке не увеличивает награду.
5. Экстремально быстрый ответ может снижать достоверность события.
6. Низкая retrievability не даёт бесконечно растущий бонус.
7. Сильная просрочка ограничивает challenge bonus.
8. Ранние reviews и cram не равны плановым reviews.
9. Массовое введение нового ограничивает immediate XP, но не отменяет будущую награду за реальное усвоение.
10. Reset, forget и повторный импорт не обновляют lifetime reward.
11. Изменение и откат не создают новый Create XP.
12. Whitespace-only и служебные массовые изменения не считаются содержательной работой.
13. Один source event не создаёт несколько полных наград.
14. Все начисления должны быть объяснимы пользователю.
15. Любая числовая формула должна проходить replay-симуляцию до реализации.

## 11. Требования к объяснимости

Для каждого начисления система должна уметь объяснить:

- какой режим сработал;
- какое исходное событие использовано;
- какая часть награды выдана сразу;
- какая часть ожидает подтверждения;
- какие коэффициенты применены;
- был ли применён repeat discount;
- был ли ограничен volume bonus;
- почему событие не получило полный XP;
- какая версия правила использована.

Пример будущего объяснения:

```text
Review XP: 7.4

Базовая награда:             5.0
Сложное воспроизведение:    ×1.25
Ожидаемый прирост памяти:   ×1.10
Время ответа достоверно:    ×1.00
Объёмный бонус:             +0.5
```

Точные числа в примере условны.

## 12. Данные и ограничения достоверности

Для модели потенциально доступны:

- идентификатор карточки;
- тип revlog-события;
- выбранная оценка;
- предыдущий и новый интервал;
- время ответа;
- learning/review/relearning/filtered classification;
- состояние карточки;
- FSRS memory state;
- note type и поля;
- изменения заметки;
- наличие media;
- leech и другие problem signals.

Перед техническим проектированием нужно отдельно подтвердить:

- какие значения FSRS доступны напрямую в поддерживаемой версии Anki;
- какие состояния требуется вычислять;
- как получить memory state до и после события;
- как надёжно отличить due, early и filtered review;
- какие editor hooks считаются стабильными;
- как идентифицировать содержательное изменение без хранения лишнего пользовательского содержимого;
- как работать с изменениями, выполненными вне обычного editor workflow;
- как вести lifetime reward state после импорта, удаления и восстановления.

## 13. Предварительно принятые решения

1. Anki является первым доменом проектирования XP.
2. XP начисляется за работу, а не за время присутствия.
3. Существуют отдельные каналы `Review`, `Learn` и `Create`.
4. Review XP зависит от состояния памяти и достоверности события.
5. Низкая retrievability повышает награду только до ограниченного оптимального диапазона.
6. Сильная просрочка не должна быть выгодной стратегией.
7. Фактическая кнопка `Easy` не создаёт прямой XP-бонус.
8. Время ответа является проверкой достоверности, а не самостоятельной валютой.
9. Объём работы даёт ограниченный boost по уникальным валидным событиям.
10. Полная награда за новую карточку распределяется во времени.
11. Большая часть Learn XP подтверждается последующим воспроизведением.
12. Create XP значительно ниже Review и Learn XP.
13. Create XP начисляется за переходы состояния, а не операции редактора.
14. Часть Create XP подтверждается последующим использованием результата.
15. Доменные антифарм-правила применяются до общего дневного diminishing returns.

## 14. Параметры-гипотезы

Пока не приняты окончательно:

- абсолютный Base Review Reward;
- все коэффициенты Review XP;
- границы retrievability;
- форма функции challenge bonus;
- метод контрфактического memory gain;
- допустимые значения response-time ratio;
- пороги volume bonus;
- скидки за same-day repeats;
- политика filtered decks и cram;
- размер Completion Bonus;
- полный Acquisition Reward;
- распределение Learn XP по этапам;
- персональная норма новых карточек;
- относительная стоимость Create XP;
- доля immediate и pending Create XP;
- дневной cap Create XP;
- формула зачёта Anki-дня для стрика.

## 15. Обязательная симуляция

Перед реализацией модель необходимо прогнать минимум на следующих сценариях.

### Review

1. Стабильная очередь с retrievability около цели.
2. Трудные, но своевременно повторяемые карточки.
3. Успешное воспроизведение при низкой retrievability.
4. Большой случайный backlog.
5. Намеренное откладывание ради challenge bonus.
6. Много `Again` и relearning steps.
7. Быстрый честный пользователь.
8. Пользователь, нажимающий ответы подозрительно быстро.
9. Пользователь, оставляющий карточки открытыми.
10. Filtered deck и cram.
11. Сотни валидных reviews за день.
12. Многократное повторение одной карточки.

### Learn

13. Устойчивое введение небольшого количества новых карточек.
14. Массовое введение сотен карточек.
15. Новые карточки, которые хорошо закрепляются.
16. Новые карточки с частыми ошибками.
17. Reset и повторное изучение ранее награждённого материала.
18. Импорт большой готовой колоды.

### Create

19. Создание качественной заметки вручную.
20. Заполнение обязательных полей.
21. Многократное редактирование одного поля.
22. Whitespace-only изменения.
23. Массовая смена тегов.
24. Исправление broken media.
25. Переработка leech-карточки.
26. Создание и быстрое удаление заметок.
27. Копирование большого количества дубликатов.
28. Отдельный день подготовки материала без reviews.

Симуляция должна показать:

- распределение XP между каналами;
- средний XP на активный день;
- влияние FSRS-сигналов;
- выгоду потенциальных exploit-стратегий;
- частоту pending и подтверждённых наград;
- влияние объёмных бонусов;
- вклад Create XP;
- скорость уровневой прогрессии;
- различия между новичком, стабильным пользователем и пользователем с backlog.

## 16. Следующий продуктовый шаг

Следующий этап — не реализация UI и не подключение к production payload.

Сначала требуется подготовить **Anki XP simulation specification**:

- выбрать нормализованную базовую единицу;
- сформировать синтетические профили;
- определить replay-наборы revlog;
- зафиксировать несколько конкурирующих формул Review XP;
- проверить кривые retrievability и memory gain;
- проверить volume bonus и same-day discount;
- смоделировать staged Learn XP;
- оценить верхний вклад Create XP;
- сопоставить итоговый дневной XP с кривой уровней.

Только после симуляции можно фиксировать числовую экономику `v1`.
