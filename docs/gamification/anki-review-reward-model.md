# Anki Review Reward Model

Статус: **DRAFT v0.1**  
Дата: **2026-07-16**  
Область: **второй этап проектирования Review XP; продуктовый концепт без production-реализации**

## 1. Назначение документа

Этот документ определяет первую проверяемую модель относительной награды за один валидный эпизод повторения в Anki.

Таксономия событий и право конкретного события участвовать в расчёте описаны в [anki-review-event-taxonomy.md](anki-review-event-taxonomy.md). Общий фундамент Anki XP описан в [anki-xp-foundation.md](anki-xp-foundation.md).

Документ отвечает на вопрос:

> Сколько условных единиц учебной награды должен получить уже классифицированный `Review Episode`, если учитывать исход воспроизведения, прогнозируемую сложность, изменение памяти, достоверность события и качество доступной модели?

На этом этапе определяется относительная стоимость отдельного эпизода. Документ не определяет:

- дневной volume bonus;
- бонус за закрытие очереди;
- общий diminishing returns;
- окончательный глобальный XP;
- стрик и `Momentum`;
- скорость уровней;
- награды за Learn XP и Create XP.

Все числовые значения являются кандидатами для симуляции, а не окончательной экономикой.

## 2. Что изменилось относительно первого foundation

Ранняя гипотеза предполагала мультипликативную формулу:

```text
Base
× Outcome
× Retrievability
× Memory Gain
× Difficulty
× Response Validity
```

Эта модель отклонена как основная.

Причина: FSRS уже связывает `Difficulty`, `Stability`, `Retrievability` и оценку ответа при вычислении нового состояния памяти. Независимое перемножение отдельных коэффициентов может несколько раз вознаградить одну и ту же характеристику карточки и создать трудно контролируемые экстремальные значения.

Второй этап принимает ограниченную аддитивную модель:

```text
честная попытка
+ успешное воспроизведение
+ ограниченный контекстный бонус
```

После сложения применяются только проверочные коэффициенты, которые могут уменьшить сомнительное событие, но не разгоняют награду.

Этот документ заменяет раннюю формулу и числовые шкалы Review XP из `anki-xp-foundation.md`. Foundation должен хранить только краткое резюме и ссылку на эту спецификацию.

## 3. Промежуточная единица Review Unit

До калибровки глобального XP используется относительная единица:

```text
1.00 Review Unit
=
обычное успешное плановое повторение
при правдоподобном поведении и нормальном контексте памяти
```

`Review Unit` не является отображаемым пользователю XP.

Преимущества промежуточной единицы:

- относительные правила можно проектировать до кривой уровней;
- формула не привязана к произвольному масштабу вроде `1`, `5` или `20 XP` за карточку;
- симулятор сможет отдельно подобрать коэффициент конвертации в глобальный XP;
- разные версии модели можно сравнивать без миграции всей экономики.

## 4. Инварианты награды

1. `Again` не даёт ноль, поскольку честная попытка извлечения знания является работой.
2. `Hard`, `Good` и `Easy` не создают разные прямые игровые награды.
3. Низкая `Retrievability` повышает награду только при успешном воспроизведении.
4. Намеренная просрочка не должна быть оптимальной XP-стратегией.
5. `Difficulty` не имеет отдельного положительного множителя.
6. Время ответа не создаёт положительный бонус; оно используется для проверки достоверности.
7. Фактическое нажатие `Easy` не увеличивает игровой memory-gain bonus.
8. Отсутствие FSRS не лишает пользователя базовой награды.
9. Обычное успешное повторение должно находиться около `1.00 Review Unit`.
10. Даже особенно ценное core-повторение не должно приносить в несколько раз больше обычного.
11. Каждая награда должна раскладываться на понятные причины.
12. Окончательные значения принимаются только после симуляции честных и эксплойтных сценариев.

## 5. Базовая формула core Review Episode

Для события класса `core` используется первая кандидатная формула:

```text
CoreReviewUnits =
    ResponseValidity
  × (
        AttemptCredit
      + Pass × OutcomeCredit
      + Pass × ContextCredit
    )
```

Где:

```text
Pass = 0 для Again
Pass = 1 для Hard, Good или Easy
```

Контекстный компонент:

```text
ContextCredit =
    NeutralContextCredit
  + ModelConfidence
    × (
        ChallengeCredit
      + MemoryGainCredit
      - NeutralContextCredit
    )
```

Итоговый порядок:

```text
Review Episode
→ eligibility из Event Taxonomy
→ pass/fail
→ retrieval challenge
→ backlog protection
→ counterfactual memory gain
→ model-confidence blending
→ response-validity check
→ Review Units
```

## 6. Кандидатные базовые значения

```text
AttemptCredit        = 0.25
OutcomeCredit        = 0.65
NeutralContextCredit = 0.10
```

Отсюда:

```text
Again без дополнительных support events:
0.25 Review Unit

Обычное успешное повторение без надёжной FSRS-модели:
0.25 + 0.65 + 0.10 = 1.00 Review Unit
```

### 6.1. Предварительный диапазон core-награды

| Сценарий | Ожидаемый диапазон |
|---|---:|
| Честный `Again` | около `0.25` |
| Очень лёгкое успешное повторение | `0.90–0.98` |
| Обычное успешное повторение | около `1.00` |
| Трудное успешное повторение | `1.10–1.25` |
| Особенно ценное успешное повторение | максимум около `1.30–1.32` |

Это диапазон до будущих дневных caps и агрегирования.

## 7. Attempt Credit

`AttemptCredit` — минимальная награда за валидную первичную проверку памяти.

```text
AttemptCredit = 0.25
```

Его получает любой валидный `core Review Episode`, включая `Again`.

Причины:

- пользователь действительно попытался вспомнить материал;
- проверка выявила забывание;
- scheduler получил честную оценку;
- карточка перешла к восстановлению;
- нулевая награда подталкивала бы использовать `Hard` вместо `Again`.

`AttemptCredit` не увеличивается при низкой retrievability. Неуспешная попытка не получает challenge или memory-gain bonus.

## 8. Outcome Credit

```text
OutcomeCredit = 0.65
```

Он начисляется только при успешном воспроизведении:

| Ответ | Pass | Outcome Credit |
|---|---:|---:|
| `Again` | `0` | `0.00` |
| `Hard` | `1` | `0.65` |
| `Good` | `1` | `0.65` |
| `Easy` | `1` | `0.65` |

Игровая система не утверждает, что `Hard`, `Good` и `Easy` являются одинаковыми scheduler-событиями. Она лишь не делает конкретную успешную кнопку выгоднее для XP.

Фактическая кнопка остаётся доступной:

- scheduler;
- диагностике неверного использования оценок;
- анализу качества обучения;
- будущей abuse-модели.

Но она не создаёт прямой бонус.

## 9. Retrieval Challenge

### 9.1. Назначение

`ChallengeCredit` вознаграждает успешное воспроизведение при низкой прогнозируемой вероятности вспомнить карточку.

Он не начисляется при `Again`.

### 9.2. Кандидатная кривая

Первая таблица anchor points:

| Retrievability перед ответом | Raw Challenge Credit |
|---:|---:|
| `≥ 0.95` | `0.00` |
| `0.90` | `0.05` |
| `0.80` | `0.12` |
| `0.65` | `0.22` |
| `0.50` | `0.30` |
| `0.35` | `0.30` |
| `0.20` | `0.25` |
| `0.10` | `0.15` |
| `< 0.10` | `0.10` |

Между anchor points применяется плавная интерполяция.

Форма кривой намеренно немонотонна:

```text
очень легко
→ обычная награда

умеренно трудно
→ растущий бонус

трудно, но реалистично
→ максимальный бонус

почти полное забывание
→ бонус уменьшается
```

### 9.3. Почему максимум находится не при нулевой Retrievability

Если награда непрерывно росла бы при уменьшении `R`, пользователь мог бы:

- откладывать карточки;
- копить backlog;
- снижать вероятность воспроизведения;
- затем получать максимальную награду за искусственно созданную трудность.

Система должна вознаграждать трудное успешное воспроизведение, но не запущенность как игровую стратегию.

### 9.4. Desired retention

`Retrievability` следует интерпретировать вместе с естественной точкой планового повторения.

Пользователь с более низкой desired retention выполняет меньше повторений, но каждое плановое воспроизведение может быть труднее. Система пока не запрещает этому отражаться в награде.

Однако общая доходность разных desired-retention режимов должна быть проверена симуляцией. Настройка scheduler не должна превращаться в очевидный способ увеличить XP без сопоставимого учебного результата.

## 10. Защита от backlog farming

### 10.1. Два значения Retrievability

Для просроченного события желательно знать:

```text
R_actual
=
Retrievability в момент фактического ответа

R_natural_due
=
Retrievability в естественный срок карточки
до дополнительной задержки
```

### 10.2. Базовый и дополнительный challenge

```text
C_actual = challengeCurve(R_actual)
C_due    = challengeCurve(R_natural_due)

ExtraChallenge = max(0, C_actual - C_due)
```

Просрочка не должна отнимать базовый challenge, который существовал бы в естественный срок. Ограничивается только дополнительная выгода, возникшая из-за задержки.

```text
AdjustedChallenge =
    C_due
  + ExtraChallenge × DelayCredit
```

### 10.3. Кандидатный Delay Credit

`DelayCredit` зависит от падения retrievability после естественного срока:

```text
DelayDrop = max(0, R_natural_due - R_actual)
```

| DelayDrop | DelayCredit |
|---:|---:|
| `≤ 0.05` | `1.00` |
| `0.15` | `0.85` |
| `0.30` | `0.65` |
| `0.50` | `0.45` |
| `≥ 0.70` | `0.25` |

Между точками используется интерполяция.

### 10.4. Поведение модели

- небольшая бытовая просрочка почти не меняет награду;
- умеренный backlog сохраняет часть дополнительного бонуса;
- тяжёлый backlog не обнуляет основную работу;
- намеренное ожидание имеет резко уменьшающуюся игровую отдачу;
- forced-due и manual-reschedule события остаются ограничены Event Taxonomy до применения этой формулы.

## 11. Memory Gain Credit

### 11.1. Почему используется контрфактический Good

Фактическое новое значение stability зависит от выбранной оценки.

Если использовать реальный переход после `Easy`, пользователь сможет повысить XP выбором кнопки, даже когда карточка не была действительно лёгкой.

Поэтому для всех успешных ответов используется стандартизированный сценарий:

```text
S_good_cf
=
новая Stability, которую FSRS рассчитал бы
для текущего состояния карточки при ответе Good
```

```text
Hard → hypothetical Good
Good → actual Good transition
Easy → hypothetical Good
```

### 11.2. Сигнал прироста

Кандидатная величина:

```text
RawMemoryGain = ln(S_good_cf / S_before)
```

Логарифмическое отношение предпочтительнее абсолютной разницы:

```text
S_good_cf - S_before
```

поскольку одинаковое количество дополнительных дней имеет разное значение для молодой и очень стабильной карточки.

### 11.3. Кандидатная нормализация

Первая упрощённая шкала для симуляции:

| `ln(S_good_cf / S_before)` | Memory Gain Credit |
|---:|---:|
| `≤ 0.10` | `0.00` |
| `0.25` | `0.03` |
| `0.50` | `0.06` |
| `0.80` | `0.09` |
| `≥ 1.10` | `0.12` |

Между точками используется интерполяция.

Альтернативный кандидат для симулятора:

```text
percentile-based normalization
внутри сопоставимых Stability / FSRS cohorts
```

Проект должен сравнить:

1. фиксированную логарифмическую шкалу;
2. percentile normalization;
3. модель без отдельного Memory Gain Credit.

### 11.4. Ограничение веса

```text
0.00 ≤ MemoryGainCredit ≤ 0.12
```

Memory gain имеет меньший максимальный вклад, чем retrieval challenge, поскольку оба сигнала частично описывают одну и ту же полезность повторения.

## 12. Difficulty

Отдельный `DifficultyFactor` не используется.

Причины:

- `Difficulty` уже участвует в FSRS-переходе stability;
- высокий `Difficulty` не всегда означает больше полезного усилия;
- плохая или перегруженная карточка может искусственно иметь высокую сложность;
- отдельный бонус сделал бы некачественные карточки выгодными;
- challenge и memory gain уже отражают значительную часть контекста.

`Difficulty` остаётся доступна для:

- вычисления контрфактического memory gain;
- диагностики;
- сегментации симуляции;
- будущего выявления проблемных карточек.

Но она не добавляет XP самостоятельно.

## 13. Model Confidence

### 13.1. Назначение

FSRS-контекст не всегда одинаково надёжен:

- у нового пользователя может быть мало истории;
- параметры могут быть не оптимизированы;
- событие может относиться к старым данным;
- модель памяти может быть недоступна;
- пользователь может использовать другой scheduler.

Низкая уверенность не лишает права на обычный Review XP. Она приближает контекстный бонус к нейтральному значению.

### 13.2. Кандидатные уровни

| Уровень | ModelConfidence |
|---|---:|
| `high` | `1.00` |
| `medium` | `0.60` |
| `low` | `0.25` |
| `unavailable` | `0.00` |

Формула смешивания:

```text
ContextCredit =
    NeutralContextCredit
  + ModelConfidence
    × (
        AdjustedChallenge
      + MemoryGainCredit
      - NeutralContextCredit
    )
```

При `unavailable`:

```text
ContextCredit = 0.10
```

Следовательно, обычное успешное повторение без FSRS всё равно даёт:

```text
0.25 + 0.65 + 0.10 = 1.00 Review Unit
```

## 14. Response Validity

### 14.1. Роль времени ответа

Время ответа используется как сигнал достоверности, а не источник положительного бонуса.

Неправильная механика:

```text
дольше думал
→ получил больше XP
```

Она вознаграждала бы ожидание, отвлечение и удерживание карточки открытой.

Правильная роль:

```text
правдоподобное время
→ награда не уменьшается

сомнительное время
→ награда ограничивается
```

### 14.2. Персональная ожидаемая длительность

`ExpectedResponseTime` может учитывать:

- историю конкретной карточки;
- note type;
- card template;
- наличие и длительность аудио;
- длину вопроса;
- cloze или production prompt;
- предыдущие успешные ответы;
- retrievability перед ответом.

Ожидаемая длительность должна строиться по устойчивой статистике, например по медиане или trimmed distribution, а не по среднему, чувствительному к большим паузам.

### 14.3. Кандидатные коэффициенты

| Класс поведения | ResponseValidity |
|---|---:|
| Правдоподобный диапазон | `1.00` |
| Слегка подозрительно быстро или долго | `0.90` |
| Очень быстро относительно карточки | `0.65` |
| Серия экстремально быстрых ответов | `0.25–0.50` |
| Практически невозможное событие | `0.00` |
| Долгий, но объяснимый трудной карточкой ответ | `1.00` |
| Достигнут timer cap без иных подтверждений | `0.80–0.90` |

### 14.4. Время не различает Hard, Good и Easy напрямую

Долгое время может быть дополнительным сигналом, что `Easy` выглядит сомнительно, но система не должна автоматически переписывать оценку пользователя.

Результат используется для:

- уменьшения игровой награды;
- диагностического объяснения;
- последующей abuse-модели.

Он не должен изменять scheduler Anki.

## 15. Support events после Again

Relearning events не используют core-формулу повторно.

Первая кандидатная модель:

```text
первый валидный relearning step: 0.05 Support Unit
второй валидный step:            0.04 Support Unit
завершение relearning:           0.03 Support Unit
общий cap на lapse episode:      0.12 Support Unit
```

Support reward:

- не получает retrieval challenge;
- не получает memory gain;
- не считается новым уникальным Review Episode;
- не создаёт volume bonus как отдельная карточка;
- может быть уменьшен response-validity проверкой;
- не выдаётся бесконечно при повторном `Again`.

Значения являются гипотезой для симуляции.

## 16. Supplemental events

Early review, preview и другие supplemental-события не используют полную core-формулу.

Предварительная политика:

- постоянная награда отсутствует или очень мала;
- challenge и memory-gain bonus не применяются;
- одно событие не может самостоятельно закрыть дневной стрик;
- действуют отдельные card/day caps;
- окончательные значения переходят в Review Abuse Model и Session/Day Aggregation.

Этот документ не фиксирует полный supplemental reward.

## 17. Caps отдельного эпизода

Предварительные ограничения:

```text
core Review Episode:
0.00–1.32 Review Unit

неуспешный core Episode без support:
около 0.25 Review Unit

support после lapse:
не более 0.12 Review Unit

полный lapse episode:
не более 0.37 Review Unit
до возможных validity reductions
```

`ResponseValidity = 0` может обнулить событие, если оно признано практически невозможным или техническим спамом.

## 18. Подробные примеры

Во всех примерах ниже используются кандидатные значения:

```text
AttemptCredit        = 0.25
OutcomeCredit        = 0.65
NeutralContextCredit = 0.10
```

Округление выполняется только для наглядности.

### Пример 1. Обычный своевременный Good

```text
Класс: core
R_actual: 0.90
Challenge: 0.05
Memory Gain: 0.05
Confidence: 1.00
Response Validity: 1.00
```

```text
Context = 0.10 + 1.00 × (0.05 + 0.05 - 0.10)
        = 0.10

Reward = 1.00 × (0.25 + 0.65 + 0.10)
       = 1.00 Review Unit
```

### Пример 2. Те же условия, но Hard

```text
Hard является успешным ответом.
Игровой расчёт использует тот же counterfactual Good.
```

```text
Reward = 1.00 Review Unit
```

Scheduler может назначить другой интервал, но кнопка не даёт игровой штраф или бонус сама по себе.

### Пример 3. Те же условия, но Easy

```text
Фактическое Easy не используется для Memory Gain Credit.
В расчёте применяется hypothetical Good.
```

```text
Reward = 1.00 Review Unit
```

Нажатие `Easy` ради XP бессмысленно.

### Пример 4. Честный Again

```text
Pass = 0
Response Validity = 1.00
```

```text
Reward = 1.00 × 0.25
       = 0.25 Review Unit
```

Retrievability и memory gain не увеличивают неуспешную попытку.

### Пример 5. Трудное успешное воспроизведение

```text
R_actual: 0.65
Challenge: 0.22
Memory Gain: 0.08
Confidence: 1.00
Response Validity: 1.00
```

```text
Context = 0.22 + 0.08
        = 0.30

Reward = 0.25 + 0.65 + 0.30
       = 1.20 Review Unit
```

### Пример 6. Очень трудное, но ещё рациональное воспроизведение

```text
R_actual: 0.50
Challenge: 0.30
Memory Gain: 0.10
Confidence: 1.00
Response Validity: 1.00
```

```text
Reward = 0.25 + 0.65 + 0.30 + 0.10
       = 1.30 Review Unit
```

Это близко к верхней границе core-награды.

### Пример 7. Экстремально низкая Retrievability без просрочки

Предположим, естественный срок действительно допускает очень низкую retrievability:

```text
R_actual: 0.08
Challenge: 0.10
Memory Gain: 0.12
Confidence: 1.00
```

```text
Reward = 0.25 + 0.65 + 0.10 + 0.12
       = 1.12 Review Unit
```

Экстремально низкая вероятность не создаёт максимальный challenge bonus.

### Пример 8. Умеренный backlog

```text
R_natural_due: 0.90
R_actual:      0.50
C_due:         0.05
C_actual:      0.30
DelayDrop:     0.40
DelayCredit:   примерно 0.55
```

```text
ExtraChallenge = 0.30 - 0.05
               = 0.25

AdjustedChallenge = 0.05 + 0.25 × 0.55
                  = 0.1875
                  ≈ 0.19
```

При `Memory Gain = 0.10`:

```text
Reward = 0.25 + 0.65 + 0.19 + 0.10
       = 1.19 Review Unit
```

Пользователь получает больше обычного, но меньше, чем при естественно трудном своевременном review.

### Пример 9. Тяжёлый backlog

```text
R_natural_due: 0.90
R_actual:      0.10
C_due:         0.05
C_actual:      0.15
DelayDrop:     0.80
DelayCredit:   0.25
```

```text
AdjustedChallenge = 0.05 + (0.15 - 0.05) × 0.25
                  = 0.075
```

При `Memory Gain = 0.12`:

```text
Reward = 0.25 + 0.65 + 0.075 + 0.12
       = 1.095
       ≈ 1.10 Review Unit
```

Месячная просрочка не превращает карточку в максимальную награду.

### Пример 10. Низкая уверенность FSRS

```text
Raw Challenge: 0.30
Memory Gain:   0.10
Confidence:    0.25
Neutral:       0.10
```

```text
Context = 0.10 + 0.25 × (0.30 + 0.10 - 0.10)
        = 0.175
```

```text
Reward = 0.25 + 0.65 + 0.175
       = 1.075
       ≈ 1.08 Review Unit
```

Модель замечает возможную сложность, но не доверяет ей полностью.

### Пример 11. FSRS недоступен

```text
Confidence = 0
Context = NeutralContextCredit = 0.10
```

```text
Reward = 0.25 + 0.65 + 0.10
       = 1.00 Review Unit
```

Пользователь без FSRS не теряет базовую награду.

### Пример 12. Подозрительно быстрый успешный ответ

Обычная награда до validity check:

```text
1.00 Review Unit
```

Но ответ занял время, практически невозможное для данного шаблона:

```text
ResponseValidity = 0.65
```

```text
Reward = 1.00 × 0.65
       = 0.65 Review Unit
```

### Пример 13. Очень долгий, но объяснимый ответ

```text
R_actual: 0.50
Базовая награда: 1.30
Ответ дольше персональной нормы,
но укладывается в диапазон трудного воспроизведения.
ResponseValidity: 1.00
```

```text
Reward = 1.30 Review Unit
```

Долгое время не создаёт бонус, но и не штрафуется без оснований.

### Пример 14. Timer cap

```text
Базовая награда: 1.20
ResponseValidity: 0.85
```

```text
Reward = 1.20 × 0.85
       = 1.02 Review Unit
```

Система признаёт возможную работу, но ограничивает уверенность в измерении времени.

### Пример 15. Again и нормальное relearning

```text
Core Again:                 0.25
Первый relearning step:     0.05
Второй relearning step:     0.04
Completion support:         0.03
```

```text
Total lapse episode = 0.25 + 0.05 + 0.04 + 0.03
                    = 0.37 Review Unit
```

Повторные шаги не создают несколько полноценных core-наград.

### Пример 16. Многочисленные Again на relearning

```text
Core Again: 0.25
Support events пытаются накопить 0.24
Support cap: 0.12
```

```text
Total = 0.25 + min(0.24, 0.12)
      = 0.37 Review Unit
```

Фарм на одной забытой карточке ограничен.

### Пример 17. Due-карточка в filtered deck с rescheduling

Event Taxonomy классифицирует событие как `core`.

При обычном контексте:

```text
Reward = 1.00 Review Unit
```

Сам filtered deck не уменьшает награду, если карточка действительно была due и scheduling сохраняется.

### Пример 18. Filtered preview без rescheduling

Event Taxonomy классифицирует событие как `supplemental` или `none`.

```text
Core formula не применяется.
Permanent Review XP: 0 или отдельный малый Practice Credit.
```

Точное значение будет определено в abuse/session этапах.

### Пример 19. Future-карточка принудительно сделана due

Event Taxonomy присваивает `forced_due` и не допускает обычный `core` reward.

```text
Core Review Units: не начисляются.
Допустим только будущий supplemental policy.
```

### Пример 20. Sibling был показан ранее сегодня

Событие остаётся `core`, но получает контекстную метку `sibling_exposed`.

На текущем этапе:

```text
обычная формула применяется без автоматического штрафа.
```

Будущий симулятор должен проверить, нужен ли небольшой cap на challenge bonus второго sibling.

### Пример 21. Массовое FSRS-rescheduling сделало карточку due

Служебная запись rescheduling:

```text
Reward = 0
```

Последующее реальное повторение:

```text
может получить core reward,
но schedule_origin сохраняется,
а искусственно созданный overdue bonus ограничивается.
```

### Пример 22. Пользователь отменил ответ через Undo

Предварительно показанная награда:

```text
provisional
```

После Undo:

```text
Reward = 0
```

Связанные support events и ledger-записи также должны быть отменены.

## 20. Explainability contract

Каждое начисление должно уметь объяснить себя структурировано.

Минимальный breakdown:

```text
Event class: core
Outcome: pass
Attempt credit: +0.25
Outcome credit: +0.65
Challenge credit: +0.22
Memory gain credit: +0.08
Model confidence adjustment: 1.00
Response validity: 1.00
Final Review Units: 1.20
```

Для backlog:

```text
Raw challenge: +0.30
Natural-due challenge: +0.05
Delay credit: 0.55
Adjusted challenge: +0.19
```

Для пользователя объяснение может быть сокращено:

```text
+1.20 Review Units
Трудное успешное воспроизведение
```

Но внутренний ledger должен хранить полный breakdown и версию правил.

## 21. Версионирование

Каждый расчёт должен сохранять:

```text
reward_model_version
challenge_curve_version
memory_gain_version
response_validity_version
fsrs_version_or_scheduler_context
parameter_snapshot_id
```

Изменение формулы не должно молча переписывать уже подтверждённую награду.

Для симуляции допускается повторный расчёт в отдельном scenario ledger без изменения пользовательского permanent XP.

## 22. Что должен проверить симулятор

### 22.1. Базовый баланс

- медианная награда обычного due-review;
- 10-й, 50-й, 90-й и 99-й percentile Review Units;
- доля карточек около `1.00`;
- доля карточек у cap;
- средняя награда за `Again` с relearning;
- распределение наград по maturity и stability.

### 22.2. FSRS-сигналы

- не учитывается ли `Retrievability` дважды;
- не доминирует ли Memory Gain;
- как меняется награда по `Difficulty` без отдельного бонуса;
- что происходит у новых пользователей;
- насколько экономика зависит от версии FSRS;
- стабильна ли модель после оптимизации параметров.

### 22.3. Поведение scheduler

- разные desired retention;
- обычная очередь и backlog;
- небольшие бытовые просрочки;
- длительные пропуски;
- массовое rescheduling;
- filtered decks;
- early review;
- siblings.

### 22.4. Abuse-сценарии

- намеренное ожидание;
- систематическое нажатие `Easy`;
- использование `Hard` вместо `Again`;
- экстремально быстрые ответы;
- удерживание карточки открытой;
- повторение одной карточки;
- изменение desired retention ради дохода;
- reset и forced due;
- множественные relearning steps.

## 23. Параметры-гипотезы

Не считаются окончательными:

- `AttemptCredit = 0.25`;
- `OutcomeCredit = 0.65`;
- `NeutralContextCredit = 0.10`;
- challenge curve;
- максимальный challenge `0.30`;
- максимальный memory gain `0.12`;
- backlog `DelayCredit`;
- confidence `1.00 / 0.60 / 0.25 / 0.00`;
- response-validity bands;
- support-event values;
- core cap около `1.32`;
- lapse cap около `0.37`.

## 24. Предварительно принятые решения

1. Используется `Review Unit`, а не окончательный XP за карточку.
2. Core-награда строится аддитивно.
3. `Again` получает только Attempt Credit и последующий ограниченный support.
4. `Hard`, `Good` и `Easy` не имеют прямой разницы игрового XP.
5. Challenge bonus имеет максимум и снижается при экстремально низкой `R`.
6. Дополнительная выгода backlog дисконтируется.
7. Memory gain рассчитывается через counterfactual `Good`.
8. Memory gain имеет меньший вес, чем challenge.
9. `Difficulty` не является отдельным XP-компонентом.
10. Время ответа не даёт положительного бонуса.
11. Model Confidence влияет только на контекст, а не на право получить базовую награду.
12. Без FSRS успешное обычное повторение остаётся около `1.00 Review Unit`.
13. Relearning не повторяет core-формулу.
14. Каждый расчёт обязан быть объясним и версионирован.
15. Окончательные числа принимаются после симуляции.
