# Режимы работы ChatGPT и Codex

Снимок правил: **2026-07-23**.

Этот документ определяет два разных режима работы с репозиторием:

- **ChatGPT mode** — работа в обычном чате ChatGPT с GitHub connector и при
  необходимости с командами, которые владелец запускает локально;
- **Codex mode** — автономная работа агента в локальном checkout репозитория.

Режим выбирается по фактической среде выполнения, а не по названию модели. Агент
не должен незаметно смешивать полномочия двух режимов.

Подробности:

- [ChatGPT work mode](chatgpt-work-mode.md);
- [Codex work mode](codex-agent-rules.md);
- [Codex local WSL environment](codex-local-environment.md);
- [UI prototyping and visual acceptance](ui-prototype-visual-acceptance.md).

## Выбор режима не является постоянным default

Активный режим выбирает владелец для конкретной задачи. Выбор может быть временным
и зависеть от доступного Codex quota, нужной автономности или типа работы.

Например, проект может временно выполнять программирование через Codex, пока
доступны лимиты, а затем полностью вернуться к ChatGPT mode с GitHub connector и
локальной консолью владельца. Такое переключение не требует переписывать
репозиторные контракты и не делает один режим предпочтительным навсегда.

Environment profile описывает только то, **как** выполнять задачу после выбора
режима. Он не является маркером текущего активного режима.

## Общие инварианты

Оба режима обязаны:

1. Начинать с актуальных `README.md`, `docs/ai-handoff.md`, профильных документов,
   production code и тестов нужного scope.
2. Использовать приоритет источников:

   ```text
   current production code and tests
   → current README and focused docs
   → fresh reports and artifacts
   → older plans and messages
   → assumptions
   ```

3. Не утверждать, что файл, код, лог или artifact изучен, если он не был открыт.
4. Не расширять задачу соседними product stages, cleanup или refactoring без
   необходимости для корректного результата.
5. Не дробить уже названный этап на новые вложенные подэтапы, буквенные ветки или
   бесконечные лестницы нумерации. Существующая roadmap-гранулярность является
   верхней границей декомпозиции.
6. Вести одну связную реализацию как одну задачу. Для одного этапа или одного
   remediation по умолчанию используются одна рабочая ветка и один основной PR.
7. Не перепроверять неизменившийся факт или тот же exact SHA без новой причины.
8. Не менять production code только ради устаревшего теста или harness assertion.
9. Не ослаблять security, token, sanitizer, media, action allowlist, loopback,
   privacy или package boundaries.
10. Не коммитить generated/runtime outputs, tokens, profiles, screenshots, logs,
    cache, `.ankiaddon` или E2E artifacts.
11. Для крупного UI или нового reference workspace выполнять prototype-first
    процесс: standalone prototype, независимое ревью, bounded revision и явная
    owner visual acceptance до production implementation. Cards задаёт общий UI
    foundation, но его page-specific layout не копируется механически.

## Принцип достаточного результата

Проекту нужен надёжный локальный add-on, а не бесконечная сертификационная
кампания. Завершение задачи определяется её заявленными completion criteria и
реальными блокерами.

Не являются основанием для бесконечного продолжения:

- косметическая возможность сделать код ещё красивее;
- повторная проверка уже зелёного неизменившегося контура;
- старый historical drift вне текущего scope;
- желание довести вспомогательный harness до абстрактного совершенства;
- отсутствие ещё одного необязательного отчёта, PR или workflow.

Повторное открытие завершённого результата оправдано при найденном риске
безопасности, потери данных, поломке package/public contract, подтверждённой
production regression или явном owner rejection.

## Ветки, PR и GitHub Actions

- Один coherent stage/remediation не должен порождать отдельные trigger, status,
  controller, verification и cleanup PR.
- Временный workflow не создаётся только потому, что текущему чату неудобно
  выполнить shell-команду.
- GitHub Actions не используется как удалённый терминал, редактор веток или
  универсальный debugger.
- Для routine Git operations, локальной диагностики и локальных тестов
  используются Codex checkout либо доступная владельцу WSL/PowerShell-консоль.
- Permanent workflows применяются только для их штатной роли: CI, release,
  package handoff и real-Anki integration gates.
- Docs/cleanup, непосредственно относящиеся к задаче, входят в основной PR либо в
  один финальный механический closeout. Отдельная цепочка PR для каждого файла
  запрещена.

## Проверки и stop-loss

Проверки выбираются по `test-matrix.md` и `verification-run-policy.md`.

- Cheap focused checks можно выполнять по мере необходимости.
- Docker real-Anki E2E является integration gate, а не пошаговым debugger.
- Targeted E2E запускается после локализации риска, а full E2E — после готовности
  целого кандидата.
- Не запускать повторно successful same-SHA gate без изменения кандидата или
  нового диагностического основания.
- После двух одинаковых или смежных harness failures без новой информации работа
  останавливается: фиксируется root cause, выбирается локальная диагностика либо
  честный `Paused/Incomplete`.
- Один исправленный кандидат обычно получает один final full E2E. Дополнительный
  full run допустим только после конкретного исправления выявленного blocker и
  явного обоснования; владелец может отдельно разрешить более широкий прогон.
- Failure вспомогательного harness не объявляется production failure без
  подтверждения по коду, API, logs или artifacts.

## Выбор режима

Использовать **ChatGPT mode**, когда:

- работа идёт в обычном чате ChatGPT;
- доступ к репозиторию идёт через GitHub connector;
- локальные команды может выполнять владелец;
- нужен продуктовый анализ, аудит, управление PR или ограниченная реализация.

Использовать **Codex mode**, когда:

- агент имеет локальный checkout и shell;
- требуется многофайловая реализация, refactor, тестирование или Git workflow;
- агент должен автономно создать commits, push и PR;
- задача удобнее и надёжнее выполняется непосредственно в рабочем дереве.

Если ChatGPT-задача превратилась в длительную многофайловую shell-работу, следует
перейти в Codex mode либо использовать консоль владельца, а не строить временную
GitHub Actions инфраструктуру.

## Обязательный итог

Финальный отчёт любого режима отделяет:

- что подтверждено;
- что изменено;
- какие проверки прошли;
- что не запускалось и почему;
- какие реальные ограничения или риски остались;
- branch, commits и PR, если Git использовался.

Не выдавать будущую или фоновую работу за уже выполненную.
